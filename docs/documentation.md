**ТЕХНИЧЕСКОЕ ЗАДАНИЕ**

Аналитический дашборд ресторанной сети

_FoodMiniApp - Restaurant Analytics Dashboard_

Версия 1.0 • 2025

**Назначение документа**

Данное ТЗ описывает архитектуру, логику, визуальную часть и реализацию аналитического дашборда для ресторанной сети на базе Vue 3 + shadcn-vue + Tailwind CSS. Источники данных: iiko OLAP API и PremiumBonus API.

Стек: Vue 3 • shadcn-vue • Tailwind CSS • Lucide Icons • Pinia • Vue Router

# **1\. Общее описание системы**

## **1.1 Цель и назначение**

Аналитический дашборд - отдельное SPA-приложение (Vue 3), предназначенное для мониторинга ключевых показателей ресторанной сети в режиме реального времени. Дашборд обеспечивает руководство и операционный менеджмент актуальными маркетинговыми и операционными данными с возможностью сравнения в разрезе периодов (LFL) и подразделений.

## **1.2 Ключевые принципы**

- Отдельное SPA-приложение, независимое от основного FoodMiniApp Admin Panel
- Данные агрегируются на бэкенде (Node.js/Express), фронт получает готовые ответы
- Без кэширования и записи в БД на текущем этапе
- Поддержка светлой и тёмной темы через CSS-переменные Tailwind
- Модульная архитектура: каждый блок метрик - независимый Vue-компонент
- MVC-подход: Model (Pinia stores + composables), View (компоненты), Controller (API-сервисы)

## **1.3 Технологический стек**

| **Категория**    | **Технология**               | **Назначение**                      |
| ---------------- | ---------------------------- | ----------------------------------- |
| UI-фреймворк     | Vue 3 (Composition API)      | Основа приложения                   |

# **3\. Фильтры и управление периодами**

## **3.1 Выбор периода**

Компонент PeriodSelector расположен в шапке дашборда. По умолчанию - «Сегодня».

| **Режим**            | **Описание**                         | **LFL-сравнение**                         |
| -------------------- | ------------------------------------ | ----------------------------------------- |
| **Сегодня ★**        | Текущий день (по умолчанию)          | Аналогичный день прошлой недели (−7 дней) |
| **Вчера**            | Прошлый день                         | Тот же день прошлой недели                |
| **Текущая неделя**   | Пн-текущий день                      | Предыдущая неделя                         |
| **Прошлая неделя**   | Пн-Вс прошлой недели                 | Неделя до неё                             |
| **Текущий месяц**    | 1-е - текущая дата                   | Аналогичный период прошлого месяца        |
| **Прошлый месяц**    | Полный прошлый месяц                 | Аналогичный месяц прошлого года           |
| **Текущий год**      | 1 янв - текущая дата                 | Аналогичный период прошлого года          |
| **Кастомный период** | DatePicker: дата начала - дата конца | Предыдущий период такой же длины          |

_💡 Кастомный период: если выбрано 10 дней (01.06-10.06), то LFL = предыдущие 10 дней (21.05-31.05)._

## **3.2 Выбор подразделений**

Компонент DepartmentSelector - мультиселект. По умолчанию - все подразделения.

- Структура: иерархия - Город → Рестораны
- Чекбокс «Выбрать все» - выделяет/снимает всё дерево
- Данные отображаются по каждому ресторану отдельно + итоговая строка «Все»
- При добавлении нового подразделения - автоматически появляется в списке

## **3.3 Логика LFL**

Composable useLFL.ts вычисляет изменение для каждой метрики:

function calcLFL(current: number, previous: number): LFLResult {

if (previous === 0 && current > 0) return { percent: 100, direction: 'up' }

if (previous === 0 && current === 0) return { percent: 0, direction: 'neutral' }

const percent = ((current - previous) / previous) \* 100

return { percent: Math.round(percent \* 10) / 10, direction: percent > 0 ? 'up' : 'down' }

}

Компонент LFLBadge отображает результат: зелёный со стрелкой вверх / красный со стрелкой вниз / серый при 0%. Для метрик, где рост - негативный (среднее время, дисконт), цвета инвертируются через prop :inverse="true".

# **4\. Маркетинговые метрики**

## **4.1 Источник данных - iiko OLAP API**

Бэкенд запрашивает iiko OLAP отчёт «Продажи» с группировкой по OrderType. Ключевые поля:

- DishDiscountSumInt - выручка (сумма после скидки)
- DishSumInt - сумма до скидки (для расчёта дисконта)
- UniqOrderId.OrdersCount - количество уникальных заказов
- OrderType - тип заказа: Доставка / Зал / Самовынос / Яндекс Еда

## **4.2 Каналы продаж**

Выручка, количество заказов и средний чек разбиваются по четырём каналам. Расчёт среднего чека:

averageCheck = DishDiscountSumInt / UniqOrderId.OrdersCount

Отображение: группа из 4 карточек на канал (Доставка / Зал / Самовынос / Яндекс). Каждая карточка содержит значение + LFL-бейдж. Ниже - сводная строка «Итого по всем каналам».

## **4.3 Дисконт**

- Абсолютное значение дисконта: DishSumInt − DishDiscountSumInt
- Процент дисконта: (дисконт / DishSumInt) × 100

_⚠ Для метрики % дисконта: LFL-бейдж инвертирован - рост дисконта считается негативным (красный)._

## **4.4 Клиентская база - PremiumBonus API**

| **Метрика**       | **Источник**     | **Логика вычисления**                                                                                                     |
| ----------------- | ---------------- | ------------------------------------------------------------------------------------------------------------------------- |
| **Активная база** | PremiumBonus API | Клиенты с ≥1 заказом за последние 30 дней. Endpoint: /customers/filter с параметром lastOrderDate ≥ now−30d               |
| **Новые клиенты** | PremiumBonus API | Клиенты, у которых дата первого заказа попадает в выбранный период. Поле: firstOrderDate в диапазоне \[dateFrom, dateTo\] |

_💡 Если PremiumBonus API не предоставляет прямой агрегации, бэкенд делает постраничную выборку клиентов с фильтрацией на своей стороне. Необходимо уточнить лимиты и rate limits API._

# **5\. Операционные метрики**

## **5.1 Временные показатели**

| **Метрика**                         | **Поля iiko**                          | **Логика**                                                                                                                                       |
| ----------------------------------- | -------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------ |
| **Среднее время доставки**          | OrderTime.AverageOrderTime             | Фильтр: Delivery.ServiceType = 'Доставка'. Время от создания до закрытия заказа                                                                  |
| **Среднее время самовыноса**        | OrderTime.AverageOrderTime             | Фильтр: Delivery.ServiceType = 'Самовынос'. Время от создания до выдачи                                                                          |
| **Среднее время зала**              | CloseTime − OpenTime (Зал)             | Фильтр OrderType = Зал. Поле OpenTime → CloseTime. Исключаются незакрытые заказы                                                                 |
| **Среднее время приготовления**     | Delivery.CookingFinishTime − OpenTime  | Для каналов Доставка + Самовынос. Агрегация по среднему                                                                                          |
| **Простой на тепловом столе**       | Delivery.ReadyTime − CookingFinishTime | Время между готовностью блюда и отправкой курьера. Если поле ReadyTime не доступно, отображается заглушка с пометкой 'Уточнить поле в iiko OLAP' |
| **Среднее время доставки (курьер)** | Delivery.WayDurationAvg                | Время в пути курьера от ресторана до клиента                                                                                                     |

_⚠ Для всех временных метрик: LFL-бейдж инвертирован. Уменьшение времени - позитивная динамика (зелёный). Значения отображаются в формате «ЧЧ:ММ»._

## **5.2 Фудкост**

- Источник: iiko OLAP отчёт «Себестоимость»
- Поле: ProductCostBase.Percent
- Отображение: % с двумя десятичными знаками + LFL-бейдж (инвертированный)
- Визуализация: горизонтальный прогресс-бар с цветовым порогом (зелёный &lt; 30%, жёлтый 30-35%, красный &gt; 35%)

_💡 Пороговые значения должны быть настраиваемыми в PlansStore._

## **5.3 Анализ маршрутов курьеров**

Цель: показать эффективность логистики. Маршрут - это курьерская доставка, включающая N заказов за один выезд. Агрегация на бэкенде по Delivery.CourierRoute или по связке курьер+время выезда.

| **Показатель**                 | **Условие**       | **Отображение**                                           |
| ------------------------------ | ----------------- | --------------------------------------------------------- |
| **Маршруты с 1 заказом**       | ordersInRoute = 1 | Количество + % от всех маршрутов (красный - неэффективно) |
| **Маршруты с 2 заказами**      | ordersInRoute = 2 | Количество + % (жёлтый)                                   |
| **Маршруты с 3+ заказами**     | ordersInRoute ≥ 3 | Количество + % (зелёный - эффективно)                     |
| **Среднее заказов на маршрут** | avg               | Итоговый показатель эффективности + LFL                   |

Визуализация: DonutChart с тремя сегментами (1 заказ / 2 заказа / 3+ заказов) + легенда с процентами.

_⚠ Необходимо проверить в iiko OLAP наличие поля CourierRoute или аналога. Если поле отсутствует, бэкенд агрегирует по курьеру и временному окну выезда. Этот момент требует тестирования._

# **6\. Выполнение плана**

## **6.1 Управление планами**

Вместо интеграции с Google Sheets предлагается встроенное управление планами в самом дашборде. Это надёжнее, не создаёт внешних зависимостей и доступно без авторизации в Google.

- Отдельный раздел PlansView.vue (/plans)
- CRUD для плановых значений: создание, редактирование, удаление
- Планы задаются по периодам: месяц / квартал / год
- Планы задаются по метрикам и по подразделениям (или «все»)
- Хранение: бэкенд сохраняет в БД или JSON-файле (определяется на этапе реализации)
- API: REST-эндпоинты /api/plans (GET, POST, PUT, DELETE)

## **6.2 Отображение на дашборде**

Для каждой ключевой метрики (выручка, количество заказов по каналам) на карточке отображается:

- Текущее значение
- Плановое значение за период
- % выполнения плана с прогресс-баром
- Цветовая индикация: < 70% - красный, 70-90% - жёлтый, ≥ 90% - зелёный

## **6.3 Структура данных плана**

interface Plan {

id: string

departmentId: string | 'all'

metric: 'revenue_delivery' | 'revenue_hall' | 'revenue_pickup' | ...

period: { year: number; month?: number; quarter?: number }

targetValue: number

createdAt: string

}

# **10\. Состояния загрузки и ошибок**

## **10.1 Skeleton-загрузка**

Пока данные загружаются, MetricCard отображает shadcn-vue &lt;Skeleton&gt; анимацию вместо значений. Графики показывают пустой контейнер с Skeleton-строками.

&lt;Card&gt;

&lt;Skeleton class="h-4 w-24 mb-2" /&gt; &lt;!-- название --&gt;

&lt;Skeleton class="h-8 w-32 mb-1" /&gt; &lt;!-- значение --&gt;

&lt;Skeleton class="h-5 w-16" /&gt; &lt;!-- LFL-бейдж --&gt;

&lt;/Card&gt;

## **10.2 Обработка ошибок**

- Если бэкенд вернул ошибку - карточка показывает иконку AlertCircle и текст «Ошибка загрузки»
- Кнопка Retry для повторного запроса
- Если данных за период нет (пустой ответ) - показывается «-» вместо значения, LFL = нет данных
- Toast-уведомление при сетевой ошибке (shadcn-vue &lt;Sonner&gt; или &lt;Toast&gt;)

## **10.3 Состояние «нет данных»**

Если iiko не вернул данные за предыдущий период (LFL), LFL-бейдж не отображается. Если нет данных ни за текущий период - значение метрики = «-». Если подразделение было открыто в текущем периоде, но отсутствовало в предыдущем, LFL = +100%.

## **11.2 usePeriod composable**

// composables/usePeriod.ts

export function calcLFLRange(preset: PeriodPreset, range: DateRange): DateRange {

const diffDays = dayjs(range.to).diff(range.from, 'day') + 1

switch (preset) {

case 'today': return { from: sub(range.from, {days:7}), to: sub(range.to, {days:7}) }

case 'week': return { from: sub(range.from, {weeks:1}), to: sub(range.to, {weeks:1}) }

case 'month': return { from: sub(range.from, {months:1}), to: sub(range.to, {months:1}) }

case 'year': return { from: sub(range.from, {years:1}), to: sub(range.to, {years:1}) }

case 'custom': return { from: sub(range.from, {days:diffDays}), to: sub(range.to, {days:diffDays}) }

}

}
