Ниже — полноценная инструкция по улучшению проекта `panda-dashboard`: как привести в порядок продуктовую логику, UI/UX, отчётность, кодовую базу и процесс разработки. Основание для этого плана такое: репозиторий уже содержит `apps`, `backend` и `docs`, но у него пока нет описания и релизов, а в документации проект описан как аналитический Vue 3 SPA на `shadcn-vue` и `Tailwind` для ресторанной сети. ([GitHub][1])

# Полноценная инструкция по развитию проекта

## 1. Цель

Превратить проект из “набора полезных аналитических экранов” в цельную аналитическую платформу, где:

- пользователь быстро понимает, что происходит в бизнесе;
- видит отклонения и причины;
- может провалиться в детали;
- доверяет данным;
- удобно работает и на desktop, и на mobile/tablet;
- команда понимает, какие модули приоритетны и в каком состоянии находятся.

Итоговый ориентир: не просто больше отчётов, а **более понятная, надёжная и управляемая система**.

---

## 2. Текущая отправная точка

Сейчас проект уже выглядит как монорепа с отдельными `apps`, `backend` и `docs`; в репозитории нет заполненного описания, README и релизов. Документация описывает широкий аналитический контур и SPA на Vue 3 с `shadcn-vue`, `Tailwind`, `Pinia` и `Vue Router`. ([GitHub][1])

Это означает:

- фундамент уже есть;
- проект не на стадии “с нуля”;
- главная задача теперь — **не распыляться**, а выровнять продукт и довести ключевые сценарии до сильного уровня.

---

## 3. Главные принципы дальнейшего развития

### 3.1. Меньше экранов как отдельных островов, больше единой логики

Каждый отчёт должен отвечать на один конкретный вопрос бизнеса и вести пользователя дальше:

- обзор;
- проблема;
- причина;
- действие.

### 3.2. Сначала polish ключевых сценариев, потом расширение

Не добавлять новые разделы, пока не будут выровнены основные:

- Dashboard
- Revenue / Sales
- Delivery / SLA
- Clients
- Stop List / Availability

### 3.3. Единый UX-каркас

Все аналитические страницы должны использовать одинаковую структуру:

1. Заголовок и контекст
2. Общие фильтры
3. KPI summary
4. Основной график / основной список
5. Вторичные блоки
6. Таблица / drill-down

### 3.4. Ясный уровень доверия к данным

На каждом экране пользователь должен видеть:

- насколько данные свежие;
- источник данных;
- полный ли это набор данных;
- есть ли деградация или ограничения.

## 4. Как изменить продуктовую архитектуру

## 4.1. Перестроить навигацию

### Что нужно сделать

Сгруппировать боковое меню не по отдельным отчётам, а по доменам.

### Новая структура меню

**1. Обзор**

- Dashboard
- Executive Summary

**2. Операции**

- Delivery SLA
- Delivery Delays
- Courier KPI
- Courier Map
- Stop List

**3. Продажи**

- Revenue
- Hourly Sales
- Top Dishes
- ABC Analysis
- Promotions

**4. Клиенты и маркетинг**

- Clients
- Client Analytics
- Marketing Sources

**5. Финансы и планирование**

- Foodcost
- Plans
- Forecast / Load Prediction

### Правила

- редкие или второстепенные страницы уводить в подгруппы;
- на mobile sidebar открывать через sheet/drawer;
- в меню показывать статус раздела:
  - Ready
  - Beta
  - Partial

---

## 4.2. Выделить уровни важности экранов

### Tier 1 — критические

Это экраны, которые должны быть самыми качественными и понятными:

- Dashboard
- Revenue
- Delivery SLA
- Stop List
- Clients

### Tier 2 — важные, но не первые

- Marketing Sources
- Top Dishes
- Delivery Summary
- Delivery Delays
- Foodcost

### Tier 3 — дополнительные

- Courier Map
- Promotions
- Forecast
- Plans

### Что это меняет

Любые дизайнерские, технические и продуктовые улучшения сначала делаются для Tier 1.
Только после этого масштабируются на остальные.

## 5. Полная инструкция по UI/UX

## 5.1. Ввести единый шаблон аналитической страницы

Каждая страница должна строиться по одной схеме.

### Блок 1. Header

Содержит:

- название страницы;
- краткое описание, что показывает экран;
- time freshness;
- кнопку обновления;
- действия экспорта, если нужны.

### Блок 2. Global Filters

Одинаковый фильтровый бар для всех страниц:

- период;
- подразделение / точка;
- канал / статус / источник — по ситуации;
- reset filters.

### Блок 3. KPI Summary

3–6 карточек с главными цифрами.

Пример для Clients:

- уникальные клиенты
- новые
- повторные
- спящие
- выручка
- средний чек

### Блок 4. Main Insight

Один главный график или визуальный блок:

- revenue trend
- SLA by day
- client growth
- top items
- ABC distribution

### Блок 5. Supporting Blocks

1–2 дополнительных блока:

- breakdown по источникам;
- сравнение подразделений;
- сегменты;
- причины просадки.

### Блок 6. Table / Drill-down

Нижний блок:

- подробная таблица;
- фильтрация;
- переход в детали.

---

## 5.2. Адаптивность и responsive-поведение

### Desktop

- 12-column grid
- summary cards: 4–6 колонок
- main chart + side block
- таблица на всю ширину нижним блоком

### Tablet

- 2-column summary
- графики stacked
- фильтры в collapsible panel или sheet
- таблицы упрощённые

### Mobile

- summary cards: 1 колонка
- фильтры в drawer
- таблицы заменять карточками там, где слишком много колонок
- детали строки открывать снизу через drawer

### Обязательные правила

- все таблицы должны жить внутри scroll area;
- длинные фильтры нельзя держать постоянно открытыми на mobile;
- horizontal overflow допустим только внутри таблицы, а не на всей странице;
- sticky header для таблиц и фильтров обязателен на длинных страницах.

---

## 5.3. Единая визуальная семантика

### Статусы

Нужно зафиксировать цвета:

- зелёный — норма
- жёлтый — внимание
- красный — критично
- серый — нет данных / ограниченные данные
- синий — информационно

### Применение

Эта семантика должна использоваться везде одинаково:

- SLA
- Stop List
- KPI
- Summary cards
- Trends
- Badges
- Alerts

### Важно

Один и тот же смысл не должен иметь разные цвета на разных страницах.

---

## 5.4. Поведение фильтров

### Что внедрить

- общий `FilterBar` компонент;
- единая логика query params;
- сохранение фильтров в URL;
- reset до дефолта;
- skeleton/loading для refetch.

### Правило UX

Фильтры не должны каждый раз сбрасываться при переходе назад или между близкими экранами.

## 6. Полная инструкция по отчётности

Сейчас у проекта уже широкий аналитический контур, но дальше нужно повышать не только количество, а прежде всего качество и применимость отчётов. ([GitHub][2])

## 6.1. Ввести стандарт качества отчёта

Каждый отчёт обязан содержать:

### 1. Назначение

Что именно показывает и для кого он нужен.

### 2. Источник данных

Откуда берутся данные.

### 3. Актуальность

- real-time
- near real-time
- daily
- historical

### 4. Степень полноты

- full
- partial
- fallback
- estimated

### 5. Методика

Как считается каждая метрика.

### 6. Drill-down

Куда пользователь может провалиться дальше.

### 7. Действие

Какое решение можно принять на основе отчёта.

---

## 6.2. Перевести отчёты в продуктовые пакеты

Вместо восприятия “каждый экран отдельно” нужно собрать их в пакеты.

### Delivery Pack

- Delivery Summary
- SLA
- Delays
- Courier KPI
- Courier Map
- Stop List

### Sales Pack

- Revenue
- Hourly Sales
- Top Dishes
- ABC Analysis
- Promotions

### Customer Pack

- Clients
- Client Analytics
- Marketing Sources

### Finance Pack

- Foodcost
- Plans
- Forecast

---

## 6.3. Добавить trust layer на все экраны

На каждом отчёте должен быть блок с:

- updated at;
- data source;
- data coverage;
- warnings.

### Пример

“Обновлено 14:35, источник: iiko Cloud API, покрытие: 94%, часть заказов без координат.”

Это резко повышает доверие к системе.

---

## 6.4. Внедрить drill-down вместо разрастания экранов

### Принцип

Если отчёт можно раскрыть из другого отчёта, лучше раскрывать, чем создавать новый верхнеуровневый экран.

### Примеры

- Revenue → по точкам → по часам
- Clients → по сегментам → карточка клиента
- Stop List → категория → товар → история статуса
- Top Dishes → ABC group → позиция → подразделения

---

## 7. Полная инструкция по backend и логике

## 7.1. Привести backend к единому API-стандарту

### Каждый endpoint должен возвращать:

- `data`
- `meta`
- `generatedAt`
- `source`
- `isPartial`
- `warnings`

### Пример

```json
{
  "data": {},
  "meta": {
    "generatedAt": "2026-04-18T12:00:00Z",
    "source": "iiko-cloud",
    "isPartial": false,
    "warnings": []
  }
}
```

### Зачем

- единообразие;
- проще фронту;
- проще отладка;
- проще trust layer.

---

## 7.2. Ввести обязательную валидацию входных параметров

Для всех query/body параметров:

- период;
- подразделение;
- статус;
- page / sort / groupBy;
- includeProfile;
- refresh.

### Что нужно сделать

Добавить schema validation слой:

- zod / joi / yup / аналогичный инструмент

### Почему

Сейчас проект уже достаточно большой, чтобы не полагаться на “ну фронт ведь передаст нормально”.

---

## 7.3. Разделить бизнес-логику и интеграции

### Правило

Интеграционные сервисы не должны:

- сами решать продуктовую логику;
- сами считать бизнес-метрики;
- сами собирать UI-ready response.

### Слойность

- `IikoService` / `PremiumBonusService` — только получение данных
- `AggregationService` — расчёты
- `PresentationService` — сборка response shape
- `Controller` — только orchestration

---

## 7.4. Ввести единые кэши и политику TTL

### Нужно документировать

Для каждого модуля:

- что кэшируется;
- на сколько;
- когда инвалидируется;
- что происходит при `refresh=true`.

### Минимальный шаблон

- token cache
- raw data cache
- aggregated cache
- optional enrichment cache

---

## 8. Полная инструкция по инженерной упаковке

## 8.1. Привести репозиторий в порядок

### Обязательно сделать

1. Добавить README
2. Добавить описание репозитория
3. Описать структуру монорепы
4. Синхронизировать названия приложений и workspaces
5. Удалить или обновить устаревшие ссылки в документации

Сейчас у репозитория нет описания и релизов, а структура и naming выглядят не до конца синхронизированными. ([GitHub][1])

### README должен содержать

- что это за проект;
- архитектуру;
- стек;
- структуру папок;
- как поднять локально;
- env variables;
- список основных модулей;
- статус ключевых разделов.

---

## 8.2. Добавить матрицу зрелости функций

Создать единый документ или JSON-конфиг, где для каждого экрана будет:

- `status: ready | beta | partial | planned`
- `owner`
- `source`
- `lastReviewedAt`
- `knownLimitations`

Это должно использоваться:

- в docs;
- в меню;
- на самом экране.

---

## 8.3. Ввести quality gates

Перед merge любая фича должна пройти:

- lint
- build
- smoke checks
- snapshot или component checks для ключевых UI-блоков
- unit tests на расчётные сервисы

## 11. Порядок внедрения: 30 / 60 / 90 дней

## Этап 1 — первые 30 дней

Цель: навести порядок и отполировать основу.

### Сделать

- README и repo cleanup
- новая структура меню
- единый filter bar
- единый шаблон аналитической страницы
- адаптивность таблиц
- trust badges
- feature readiness matrix
- polish для Dashboard и Revenue

### Definition of Done

- проект понятен новому разработчику;
- навигация чище;
- ключевые страницы визуально и логически едины.

---

## Этап 2 — 31–60 дней

Цель: довести ключевые аналитические модули.

### Сделать

- Clients polished
- Stop List polished
- ABC analysis вместо ассортимента
- общий API response contract
- validation layer
- unit tests на ключевые агрегации
- mobile UX pass

### Definition of Done

- Tier 1 экраны сильные и надёжные;
- ключевые расчёты протестированы;
- фронт предсказуемо работает на основных разрешениях.

---

## Этап 3 — 61–90 дней

Цель: масштабирование без хаоса.

### Сделать

- delivery pack drill-down
- customer pack drill-down
- finance pack polish
- integration tests
- status badges in menu
- quality gates in CI
- roadmap cleanup и синхронизация docs с кодом

### Definition of Done

- продукт воспринимается как единая платформа;
- новые экраны подключаются по стандарту;

## 13. KPI успеха этого улучшения

Через 2–3 итерации проект должен показать такие признаки зрелости:

### Продуктовые

- пользователь быстрее находит нужный отчёт;
- меньше вопросов “откуда эта цифра?”;
- больше drill-down, меньше лишних страниц.

### UX

- одинаковая логика страниц;
- меньше визуального шума;
- mobile usable без боли.

### Инженерные

- новый разработчик поднимает проект без угадывания;
- критические сервисы покрыты тестами;
- фичи помечены по зрелости;
- docs синхронизированы с кодом.

---

## 14. Финальная формулировка

Главная задача проекта на следующем этапе — **не просто добавить ещё функциональность, а сделать систему цельной, объяснимой и надёжной**.

Правильный порядок такой:

1. привести в порядок навигацию и архитектуру продукта;
2. унифицировать UX и шаблоны экранов;
3. сделать отчёты прозрачными по методике и качеству данных;
4. усилить backend стандартами, валидацией и тестами;
5. только после этого расширять аналитику дальше.
