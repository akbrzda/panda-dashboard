# 📘 Регламент: Стандарты написания кода

## 0. Область применения и уровни строгости

### 0.1 Область применения

Настоящий регламент распространяется на все проекты и сервисы, разрабатываемые командой.

### 0.2 Уровни требований

- **MUST** — обязательное требование. Невыполнение блокирует принятие кода.
- **SHOULD** — строгая рекомендация. Отклонение требует письменного обоснования.
- **MAY** — опциональная рекомендация на усмотрение разработчика.
- **FORBIDDEN** — категорически запрещено при любых обстоятельствах.

---

## 1. Фундаментальные принципы кода

### 1.1 Принцип единственной ответственности

**Требование:** Каждый программный модуль (файл, класс, функция) **MUST** иметь одну и только одну причину для изменения.

**Критерии соответствия:**

- Модуль решает одну чётко определённую задачу
- Изменение бизнес-требования в одной области затрагивает только один модуль
- Название модуля точно и однозначно описывает его назначение

**Признаки нарушения:**

- Модуль содержит логику из разных предметных областей
- При изменении одного требования необходимо редактировать несколько логически не связанных частей модуля
- Невозможно дать модулю краткое (1-3 слова) и точное название

---

### 1.2 Принцип явности и предсказуемости

**Требование:** Поведение кода **MUST** быть понятным из его названия, сигнатуры и контракта без изучения реализации.

**Обязательные критерии:**

- Результат выполнения функции предсказуем по её названию и типам параметров
- Отсутствуют скрытые побочные эффекты (изменение глобального состояния, запись в БД, сетевые запросы)
- Не используются неявные зависимости от контекста выполнения

**FORBIDDEN:**

- Автоматическое изменение входных параметров функции
- Обращение к глобальным переменным внутри функций
- Неявные преобразования типов данных
- Использование абстракций, объяснение которых занимает более 10 минут

---

### 1.3 Принцип изоляции бизнес-логики

**Требование:** Бизнес-логика приложения **MUST NOT** содержать прямых зависимостей от инфраструктурных компонентов.

**Запрещённые зависимости в слое бизнес-логики:**

- HTTP-фреймворки и их абстракции (request, response, session)
- UI-фреймворки и компоненты
- HTTP-клиенты и методы прямого обращения к внешним API
- Конкретные реализации баз данных и ORM
- Библиотеки для работы с мессенджерами и уведомлениями

**Правило взаимодействия:**

- Бизнес-логика получает данные через параметры функций
- Результат работы возвращается через return
- Взаимодействие с внешними системами осуществляется через интерфейсы (Repository, Gateway, Adapter)

## 2. Метрики и ограничения размера кода

### 2.1 Размер файлов

**Обязательные ограничения:**

| Количество строк | Статус            | Требуемые действия                                              |
| ---------------- | ----------------- | --------------------------------------------------------------- |
| До 300           | Норма             | Нет дополнительных требований                                   |
| 300-500          | Условно допустимо | **MUST** строго следовать принципу единственной ответственности |
| Более 500        | Недопустимо       | **MUST** разделить на несколько файлов                          |

**Правило оценки для диапазона 300-500 строк:**
Файл допускается к приёмке только если:

- Выполняет строго одну функцию на одном уровне абстракции
- Разделение приведёт к нарушению целостности логики
- В комментарии к PR указано обоснование такого размера

**Процедура разделения при превышении 500 строк:**

1. Идентифицировать отдельные ответственности внутри файла
2. Создать отдельные файлы для каждой ответственности
3. Создать index-файл для публичного API модуля
4. Обновить все импорты в зависимых файлах
5. Провести регрессионное тестирование

---

### 2.2 Размер функций

**Обязательные ограничения:**

| Количество строк | Статус              | Требуемые действия                 |
| ---------------- | ------------------- | ---------------------------------- |
| До 25            | Норма               | Нет дополнительных требований      |
| 25-40            | Допустимо           | Оценить целесообразность разбиения |
| 40-50            | Требует обоснования | **SHOULD** разделить на подфункции |
| Более 50         | Недопустимо         | **MUST** разделить на подфункции   |

**Правило декомпозиции функций:**

- Выделить логические блоки в отдельные функции
- Присвоить каждой функции имя, отражающее её назначение
- Обеспечить единый уровень абстракции внутри каждой функции

**Допустимые исключения:**

- Конфигурационные структуры данных и таблицы маппинга
- Сложные запросы к БД, где разбиение снижает читаемость
- Исключение **MUST** быть явно задокументировано в комментарии с обоснованием

## 3. Соглашения об именовании

### 3.1 Общие требования к именам

**MUST:**

- Имя отражает назначение сущности
- Имя читается как фраза на естественном языке
- Используется английский язык для всех идентификаторов
- Применяется общепринятая терминология предметной области

**FORBIDDEN:**

- Неинформативные имена: data, tmp, obj, info, handler, manager
- Сокращения, кроме общепринятых (id, url, api, http, db)
- Транслитерация с русского языка
- Нумерация переменных: data1, data2, result2
- Однобуквенные имена (кроме индексов циклов i, j, k)

---

### 3.2 Переменные и константы

**Правило:** Имя переменной **MUST** быть существительным или именной группой.

**Требования к формату:**

- Переменные: camelCase
- Константы верхнего уровня: UPPER_SNAKE_CASE
- Приватные переменные (TypeScript/поля классов): \_camelCase

**Правило для коллекций:**

- Имена массивов и списков **MUST** быть во множественном числе
- Суффиксы Array, List, Map **FORBIDDEN** (избыточны)

---

### 3.3 Функции и методы

**Правило:** Имя функции **MUST** начинаться с глагола и отражать выполняемое действие.

**Стандартные префиксы:**

- `get` — получение значения без побочных эффектов
- `fetch` — получение данных с сетевым запросом или обращением к БД
- `calculate` — вычисление значения
- `validate` — проверка валидности данных
- `create` — создание новой сущности
- `update` — изменение существующей сущности
- `delete` / `remove` — удаление сущности
- `send` — отправка данных
- `handle` — обработка события

---

### 3.4 Логические переменные и функции

**Правило:** Имена булевых переменных и функций **MUST** начинаться с префиксов: `is`, `has`, `should`, `can`, `will`.

**Требования:**

- Имя должно читаться как вопрос с ответом да/нет
- Избегать отрицаний (не `isNotActive`, а `isActive`)

---

### 3.5 Файлы и модули

**Требования к именованию файлов:**

- Формат: kebab-case (слова через дефис)
- Имя отражает содержимое файла
- Для классов: имя совпадает с именем класса

**Суффиксы файлов по типу:**

- `.service.js` — бизнес-логика
- `.repository.js` — работа с данными
- `.controller.js` — обработка HTTP
- `.validator.js` — валидация
- `.mapper.js` — преобразование данных
- `.test.js` / `.spec.js` — тесты

---

## 4. Структура и организация кода

### 4.1 Порядок секций в файле

**Обязательный порядок:**

1. Импорты внешних библиотек
2. Импорты через внутренние алиасы (@/)
3. Относительные импорты
4. Импорты типов (TypeScript)
5. Константы
6. Приватные функции
7. Публичные функции / экспорты
8. Экспорт по умолчанию (если есть)

**Требование к группировке:**

- Между группами импортов **MUST** быть пустая строка
- Импорты внутри группы **SHOULD** быть отсортированы по алфавиту

---

### 4.2 Комментарии

**Правило применения комментариев:**
Комментарии **SHOULD** объяснять "почему", а не "что".

**Когда комментарии ОБЯЗАТЕЛЬНЫ:**

- Объяснение неочевидных бизнес-правил
- Обоснование технических решений, которые кажутся странными
- Обход известных багов в сторонних библиотеках
- Временные решения (TODO, FIXME, HACK)

**Когда комментарии ЗАПРЕЩЕНЫ:**

- Дублирование того, что очевидно из кода
- Закомментированный код
- Устаревшие комментарии, не соответствующие коду

**Формат служебных комментариев:**

```
// TODO(username): описание задачи и дедлайн
// FIXME(username): описание проблемы и её причины
// HACK(username): почему используется костыль и когда будет исправлено
```

---

## 5. Обработка ошибок

### 5.1 Общие требования

**MUST:**

- Все ошибки **MUST** быть обработаны явно
- Пустые блоки catch **FORBIDDEN**
- Каждая ошибка **MUST** быть залогирована с достаточным контекстом

**Требование к логированию ошибок:**

- Сообщение об ошибке
- Трассировка стека
- Контекст выполнения (идентификаторы сущностей, параметры запроса)
- Идентификатор пользователя (если применимо)

---

### 5.2 Типизация ошибок

**Требование:** Приложение **MUST** использовать специализированные классы ошибок, а не базовый Error.

**Обязательные типы ошибок:**

- ValidationError — ошибки валидации входных данных
- NotFoundError — запрашиваемая сущность не найдена
- UnauthorizedError — пользователь не авторизован
- ForbiddenError — недостаточно прав для операции
- ConflictError — конфликт данных (дубликат уникального поля)
- InternalError — внутренняя ошибка сервера

**Свойства класса ошибки:**

- `name` — название класса ошибки
- `message` — человекочитаемое описание
- `code` — машиночитаемый код ошибки
- `statusCode` — HTTP статус-код (для API)

---

### 5.3 Централизация обработки

**Требование:** Приложение **MUST** иметь единый глобальный обработчик ошибок.

**Функции глобального обработчика:**

- Логирование всех ошибок
- Преобразование ошибок в унифицированный формат ответа
- Скрытие внутренних деталей реализации от клиента
- Установка корректных HTTP статус-кодов

**FORBIDDEN в обработчике:**

- Раскрытие трассировок стека клиенту в production
- Раскрытие путей к файлам и внутренней структуры
- Передача технических деталей БД или внутренних систем

---

## 6. Логирование

### 6.1 Требования к логированию

**MUST:**

- Использовать единую библиотеку логирования во всём проекте
- Логировать все входящие запросы
- Логировать все ошибки с полным контекстом
- Логировать критичные бизнес-операции

**Уровни логирования:**

- `error` — ошибки, требующие немедленного внимания
- `warn` — потенциальные проблемы, не критичные
- `info` — важные бизнес-события (создание заказа, регистрация)
- `debug` — детальная информация для отладки (только dev/staging)

---

### 6.2 Требования к содержимому логов

**MUST включать:**

- Временную метку
- Уровень логирования
- Описание события
- Контекст (идентификаторы сущностей)

**FORBIDDEN логировать:**

- Пароли
- Токены доступа
- Номера банковских карт
- Персональные данные (без явного требования)
- API ключи
- Секретные ключи шифрования

---

## 7. Переменные окружения

### 7.1 Требования к структуре

**MUST:**

- Файл `.env` **MUST** быть в `.gitignore`
- Репозиторий **MUST** содержать `.env.example` с шаблоном
- Все обязательные переменные **MUST** быть перечислены в `.env.example`
- Секретные значения в `.env.example` **MUST** быть пустыми или заменены на заглушки

---

### 7.2 Валидация переменных окружения

**Требование:** Приложение **MUST** валидировать наличие всех обязательных переменных окружения при запуске.

**Действия при валидации:**

- Проверка наличия всех обязательных переменных
- Проверка формата значений (число, URL, enum)
- Прерывание запуска с понятным сообщением об ошибке

**FORBIDDEN:**

- Использовать значения по умолчанию для секретных ключей
- Запускать приложение с отсутствующими критичными переменными
- Игнорировать ошибки валидации

## 8. Контроль версий (Git)

### 8.1 Формат commit сообщений

**Обязательный формат:**

```
<type>(<scope>): <subject>

<body>
```

**Типы коммитов:**

- `feat` — новая функциональность
- `fix` — исправление бага
- `refactor` — рефакторинг без изменения поведения
- `docs` — изменения в документации
- `style` — форматирование, отступы (не влияет на логику)
- `test` — добавление или изменение тестов
- `chore` — рутинные задачи (обновление зависимостей)

**Требования к subject:**

- Длина до 72 символов
- Начинается с глагола в повелительном наклонении
- Не заканчивается точкой
- На английском языке

---

### 8.2 Требования к коммитам

**MUST:**

- Коммит **MUST** быть атомарным (одно логическое изменение)
- Коммит **MUST** оставлять код в рабочем состоянии
- Описание **MUST** точно отражать изменения

**FORBIDDEN:**

- Коммиты с нарушенными тестами
- Коммиты с неописательными сообщениями (fix, update, changes)
- Коммиты с множественными не связанными изменениями
- Коммиты с закомментированным кодом

---

## 9. Code Review

### 9.1 Чек-лист автора PR

**Перед отправкой на ревью автор MUST проверить:**

- [ ] Код соответствует принципу единственной ответственности
- [ ] Размеры файлов и функций в допустимых пределах
- [ ] Все имена информативны и следуют соглашениям
- [ ] Отсутствует дублирование кода
- [ ] Все ошибки обработаны
- [ ] Нет закомментированного кода
- [ ] Нет захардкоженных значений (используется конфигурация)
- [ ] Добавлены необходимые тесты
- [ ] Все тесты проходят
- [ ] Коммиты атомарны и хорошо описаны

---

### 9.2 Чек-лист ревьюера

**Ревьюер MUST проверить:**

- [ ] Соблюдение архитектурных принципов
- [ ] Правильность границ между слоями
- [ ] Отсутствие god-классов и god-функций
- [ ] Корректность зависимостей между модулями
- [ ] Отсутствие дублирования существующей функциональности
- [ ] Читаемость и понятность кода
- [ ] Наличие и качество тестов
- [ ] Корректность обработки ошибок

---

### 9.3 Критерии блокировки PR

**PR MUST быть заблокирован при:**

- Файл превышает 500 строк без обоснования
- Функция превышает 50 строк без обоснования
- Прямой импорт внутренностей модуля (нарушение инкапсуляции)
- Бизнес-логика находится вне Service слоя
- Отсутствует обработка ошибок
- Дублирование существующей логики
- Использование сложных абстракций без обоснования
- Закомментированный код
- Некорректные commit сообщения

---

## 10. Тестирование

### 10.1 Обязательное покрытие тестами

**MUST покрыть:**

- Аутентификация и авторизация
- Критичные бизнес-процессы
- Расчётные функции
- Валидация данных
- Обработка ошибок

**SHOULD покрыть:**

- Утилиты и вспомогательные функции
- API эндпоинты
- Преобразование данных (mappers)

---

### 10.2 Приоритеты тестирования

**Порядок приоритетов:**

1. Unit-тесты — тестирование отдельных функций и методов
2. Integration-тесты — тестирование взаимодействия модулей
3. E2E-тесты — сквозное тестирование пользовательских сценариев

**Требование к unit-тестам:**

- Тесты **MUST** быть изолированными
- Внешние зависимости **MUST** быть замокованы
- Один тест проверяет одно поведение

---

## 11. Definition of Done

**Задача считается выполненной когда:**

- [ ] Код соответствует всем требованиям регламента
- [ ] Размеры файлов и функций в допустимых пределах
- [ ] Соблюдены соглашения об именовании
- [ ] Отсутствует дублирование кода
- [ ] Обработаны все ошибки
- [ ] Добавлено логирование критичных операций
- [ ] Написаны и пройдены тесты
- [ ] Code review пройден
- [ ] Документация обновлена (если требуется)
- [ ] Нет TODO без указания сроков выполнения
