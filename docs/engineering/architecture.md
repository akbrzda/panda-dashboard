# 🏗️ Регламент: Архитектура приложения

## 0. Область применения

Настоящий регламент определяет обязательные требования к архитектуре Backend (Node.js/Express) и Frontend (Vue 3) приложений.

---

## 1. Архитектурный паттерн

### 1.1 Применяемая архитектура

**Требование:** Приложение **MUST** использовать комбинацию паттернов MVC и Layered Architecture.

**Обязательные слои (Backend):**

1. Routes — определение маршрутов
2. Controller — обработка HTTP
3. Service — бизнес-логика
4. Repository — работа с данными

**Правило направления зависимостей:**

- Зависимости **MUST** идти только сверху вниз
- Обратные зависимости **FORBIDDEN**
- Схема: Routes → Controller → Service → Repository → Database

---

### 1.2 Цели архитектуры

**Архитектура считается корректной если:**

- Изменение бизнес-требования локализовано в одном слое
- Замена инфраструктуры (БД, HTTP-фреймворк) не затрагивает бизнес-логику
- Бизнес-логику можно тестировать изолированно
- Новый разработчик понимает структуру за 15-20 минут

---

## 2. Структура Backend модуля

### 2.1 Обязательная файловая структура

**Стандарт организации модуля:**

```
modules/<domain>/
  index.js          # публичный API модуля (ОБЯЗАТЕЛЬНО)
  routes.js         # определение маршрутов
  controller.js     # обработка HTTP запросов
  service.js        # бизнес-логика
  repository.js     # работа с БД
  validators.js     # валидация входных данных
  mapper.js         # трансформация данных (DTO)
```

**Требование к index.js:**

- Файл **MUST** экспортировать только публичный API модуля
- Внутренние детали (repository, mapper, validators) **MUST NOT** быть экспортированы
- Другие модули **MUST** импортировать только из index.js

**FORBIDDEN:**

- Импорт напрямую из внутренних файлов модуля
- Экспорт приватных деталей реализации через index.js

---

### 2.2 Слой Routes

**Ответственность:**

- Определение URL маршрутов
- Связывание URL с методами контроллера
- Подключение middleware (аутентификация, валидация)

**MUST:**

- Содержать только определения маршрутов
- Использовать Router из Express
- Применять middleware в правильном порядке

**FORBIDDEN:**

- Бизнес-логика
- Прямые запросы к БД
- Обработка ошибок (кроме подключения middleware)
- Трансформация данных

---

### 2.3 Слой Controller

**Ответственность:**

- Извлечение данных из HTTP запроса (req.body, req.params, req.query)
- Запуск валидации входных данных
- Вызов методов Service
- Формирование HTTP ответа (статус-код, JSON)
- Передача ошибок в глобальный error handler

**MUST:**

- Использовать async/await для асинхронных операций
- Передавать все ошибки через next(error)
- Возвращать единообразный формат ответа

**FORBIDDEN:**

- Бизнес-логика (расчёты, проверки бизнес-правил)
- SQL запросы
- Прямая работа с БД
- Знание о внутренней структуре БД
- Отправка email, SMS, push-уведомлений
- Работа с внешними API

---

### 2.4 Слой Service

**Ответственность:**

- Вся бизнес-логика приложения
- Координация между несколькими репозиториями
- Проверка бизнес-правил и прав доступа
- Выполнение расчётов
- Принятие бизнес-решений

**MUST:**

- Содержать всю логику предметной области
- Быть независимым от HTTP и фреймворков
- Получать данные через параметры функций
- Возвращать результат через return
- Использовать Repository для доступа к данным

**FORBIDDEN:**

- Знание о req, res, next, HTTP-заголовках
- Прямые SQL запросы
- Знание о структуре БД
- Отправка HTTP статус-кодов
- Зависимость от Express или других HTTP-фреймворков

**Правило тестируемости:**
Service **MUST** быть полностью тестируемым без запуска HTTP сервера и без доступа к БД.

---

### 2.5 Слой Repository

**Ответственность:**

- Все операции с базой данных (CRUD)
- Составление SQL запросов
- Трансформация данных БД в объекты домена
- Обработка транзакций

**MUST:**

- Возвращать объекты домена, а не сырые данные из БД
- Абстрагировать логику работы с БД от остального приложения
- Обрабатывать ошибки БД

**FORBIDDEN:**

- Бизнес-логика
- Проверка бизнес-правил
- Проверка прав доступа
- HTTP логика
- Форматирование данных для API ответа

---

### 2.6 Валидаторы

**Назначение:** Проверка формата и корректности входных данных.

**Правила валидации:**

- Валидация **MUST** происходить до вызова Service
- Валидатор **MUST** возвращать структурированный список ошибок
- Применяется в Controller или как middleware

**Что проверяется:**

- Наличие обязательных полей
- Типы данных
- Форматы (email, телефон, дата)
- Диапазоны значений
- Максимальная длина строк

**Что НЕ проверяется в валидаторах:**

- Бизнес-правила (например, "заказ можно отменить только в течение 1 часа")
- Проверки на уровне БД (уникальность, существование связанных сущностей)

---

### 2.7 Mappers (DTO)

**Назначение:** Преобразование данных между слоями.

**Виды маппинга:**

- БД → Домен → API Response
- API Request → Домен → БД

**Обязательные преобразования:**

- snake_case (БД) ↔ camelCase (API)
- Типы данных БД → JavaScript типы
- Скрытие внутренних полей (пароли, служебные поля)

**MUST:**

- Быть чистыми функциями без побочных эффектов
- Не содержать бизнес-логику

---

## 3. Dependency Injection

### 3.1 Принцип внедрения зависимостей

**Требование:** Модули **MUST** получать зависимости через конструктор или параметры функций, а не создавать их напрямую.

**Цели DI:**

- Снижение связанности между модулями
- Возможность замены реализации
- Упрощение тестирования (подмена реальных зависимостей моками)

**Правило:** Service и Repository **MUST** получать свои зависимости через конструктор.

---

### 3.2 Применение DI

**Обязательное внедрение через конструктор:**

- Repository в Service
- Другие Service в Service
- Внешние сервисы (Email, SMS, Payment) в Service

**Преимущества для тестирования:**

- Возможность подменить Repository на mock
- Тестирование Service без доступа к БД
- Изоляция unit-тестов

---

## 4. Границы модулей

### 4.1 Инкапсуляция модулей

**Требование:** Каждый модуль **MUST** иметь чёткую границу и публичный API.

**Публичный API:**

- Экспортируется через index.js
- Содержит только то, что необходимо внешним модулям
- Является стабильным контрактом

**Приватные детали:**

- Repository
- Validators
- Mappers
- Вспомогательные утилиты

**FORBIDDEN:**

- Прямой импорт приватных файлов модуля
- Обход index.js при импорте
- Экспорт приватных деталей в публичный API

---

### 4.2 Взаимодействие между модулями

**Разрешённые зависимости:**

- Модуль может использовать Service других модулей через их публичный API
- Импорт только через index.js

**FORBIDDEN:**

- Прямое обращение к Repository другого модуля
- Обращение к внутренним деталям модуля
- Циклические зависимости между модулями

**Правило слабой связанности:**
Модуль **SHOULD** зависеть не более чем от 3-5 других модулей.

---

## 5. Frontend архитектура (Vue 3)

### 5.1 Структура проекта

**Обязательная структура:**

```
src/
  router/              # Vue Router конфигурация
  stores/              # Pinia stores
  services/            # Бизнес-логика (без Vue)
  api/                 # HTTP клиенты
  composables/         # Переиспользуемая логика
  components/
    ui/                # Презентационные компоненты
    features/          # Бизнес-компоненты
    layouts/           # Layout компоненты
  views/               # Страницы (роуты)
  utils/               # Утилиты
```

---

### 5.2 Composition API

**Требование:** Все новые компоненты **MUST** использовать Composition API.

**Обязательный формат:**

- `<script setup>` для всех компонентов
- Composables для переиспользуемой логики
- Pinia для глобального состояния

**FORBIDDEN:**

- Options API для новых компонентов
- Mixins (использовать composables)
- Vuex (использовать Pinia)

---

### 5.3 Разделение компонентов

#### 5.3.1 Презентационные компоненты (UI)

**Назначение:** Чистые компоненты для отображения.

**MUST:**

- Получать все данные через props
- Эмитить события через emit
- Не содержать бизнес-логику
- Не обращаться к API
- Не использовать Pinia stores

**Применение:**

- Кнопки, поля ввода, карточки
- Модальные окна, дропдауны
- Переиспользуемые UI элементы

---

#### 5.3.2 Бизнес-компоненты (Features)

**Назначение:** Компоненты с бизнес-логикой.

**MUST:**

- Использовать composables для логики
- Обращаться к Pinia stores для состояния
- Композировать UI компоненты

**Применение:**

- Карточка заказа
- Форма создания сущности
- Списки с фильтрацией

---

#### 5.3.3 Страницы (Views)

**Назначение:** Компоненты-контейнеры для роутов.

**Ответственность:**

- Композиция бизнес-компонентов
- Управление состоянием страницы
- Обработка событий навигации

---

### 5.4 Composables

**Требование:** Переиспользуемая логика **MUST** быть вынесена в composables.

**Правила именования:**

- Префикс `use`: useOrders, useAuth, useForm

**Что выносить в composables:**

- Работа с API
- Управление состоянием (reactive, ref)
- Обработка форм
- Работа с WebSocket
- Бизнес-логика компонентов

**MUST:**

- Возвращать реактивные данные и функции
- Быть независимыми от конкретных компонентов
- Быть переиспользуемыми

---

### 5.5 Pinia Stores

**Назначение:** Управление глобальным состоянием приложения.

**MUST:**

- Использовать для данных, доступных в разных частях приложения
- Содержать getters для вычисляемых свойств
- Actions для изменения состояния

**FORBIDDEN в stores:**

- Прямые HTTP запросы (использовать API layer)
- Логика роутинга (router.push)
- Бизнес-расчёты (выносить в services)
- Множественные ответственности

**Правило:** Один store = одна предметная область (auth, orders, cart, settings).

---

### 5.6 API Layer

**Требование:** Все HTTP запросы **MUST** быть изолированы в API слое.

**Структура API слоя:**

- Базовая конфигурация axios (interceptors, baseURL, timeout)
- Отдельные API модули для каждой предметной области

**Ответственность API модуля:**

- Выполнение HTTP запросов
- Обработка токенов аутентификации
- Автоматическое обновление access token
- Преобразование ошибок

**FORBIDDEN:**

- HTTP запросы напрямую из компонентов
- HTTP запросы из stores

---

### 5.7 Router Guards

**Требование:** Логика проверки доступа **MUST** быть в router guards.

**Виды guards:**

- `authGuard` — проверка аутентификации
- `roleGuard` — проверка ролей
- `permissionGuard` — проверка конкретных прав

**FORBIDDEN в guards:**

- Бизнес-логика
- WebSocket подключения
- Аналитика
- Сложные вычисления

---

## 6. Express Middleware

### 6.1 Порядок middleware

**Требование:** Middleware **MUST** быть подключены в строго определённом порядке.

**Обязательный порядок:**

1. Безопасность (helmet, cors)
2. Парсинг тела запроса (body-parser, json)
3. Логирование запросов
4. Аутентификация (где требуется)
5. Роутеры приложения
6. 404 handler
7. Error handler (последним)

**Критичность порядка:**

- helmet **MUST** быть первым
- errorHandler **MUST** быть последним
- Нарушение порядка приводит к уязвимостям или неработающей обработке ошибок

---

### 6.2 Переиспользуемые middleware

**Требование:** Кастомные middleware **MUST** быть вынесены в отдельные файлы.

**Стандартные middleware:**

```
middleware/
  auth.js           # проверка JWT токена
  validate.js       # валидация схем
  checkRole.js      # проверка ролей
  rateLimit.js      # ограничение запросов
  errorHandler.js   # глобальная обработка ошибок
```

**MUST:**

- Иметь понятное назначение
- Быть переиспользуемыми
- Обрабатывать ошибки корректно

---

## 7. Работа с базой данных

### 7.1 Миграции

**Требование:** Все изменения схемы БД **MUST** выполняться через миграции.

**Правила миграций:**

- Одна миграция = одно логическое изменение
- Миграции нумеруются последовательно
- Миграции **MUST** быть идемпотентными
- Миграции **MUST** содержать rollback логику

**Структура:**

```
migrations/
  001_create_users_table.sql
  002_create_orders_table.sql
  003_add_loyalty_points_to_users.sql
```

**FORBIDDEN:**

- Прямое изменение схемы БД в production
- Пропуск миграций
- Изменение уже применённых миграций

---

### 7.2 Проблема N+1 запросов

**Требование:** Repository **MUST** избегать N+1 проблемы.

**Методы решения:**

- Использование JOIN для загрузки связанных данных
- Использование IN для массовой загрузки
- Eager loading вместо lazy loading

**Критерий проверки:**
Количество запросов к БД **MUST** быть константным и не зависеть от размера выборки.

---

## 8. Признаки хорошей архитектуры

### 8.1 Критерии оценки

**Архитектура считается качественной если:**

**Локализация изменений:**

- Изменение тарифов доставки затрагивает только DeliveryService
- Замена MySQL на PostgreSQL затрагивает только Repository слой
- Добавление нового поля в API затрагивает Controller и Mapper

**Лёгкость добавления функциональности:**

- Новый модуль создаётся по шаблону за 15-30 минут
- Новый эндпоинт добавляется без изменения существующих
- Новая бизнес-логика добавляется в Service без затрагивания Controller

**Заменяемость зависимостей:**

- Замена Telegram на WhatsApp не требует изменения бизнес-логики
- Переход с Express на Fastify затрагивает только Controller
- Замена БД затрагивает только Repository

**Тестируемость:**

- Бизнес-логику можно тестировать без HTTP и БД
- Моки легко подменяют реальные зависимости
- Unit-тесты запускаются за секунды

---

## 9. Антипаттерны и их устранение

### 9.1 God-class / God-file

**Определение:** Файл/класс, который делает слишком много.

**Признаки:**

- Размер более 500 строк
- Содержит логику из разных предметных областей
- Имеет более 10 зависимостей
- Изменяется при любом изменении требований

**Решение:**

- Разделить по принципу единственной ответственности
- Выделить отдельные модули
- Создать чёткие границы через index.js

---

### 9.2 Смешение слоёв

**Определение:** Логика разных слоёв находится в одном файле.

**Признаки:**

- SQL запросы в Controller
- req/res в Service
- Бизнес-логика в Repository
- HTTP статус-коды в Service

**Решение:**

- Переместить логику в соответствующий слой
- Создать чёткие интерфейсы между слоями

---

### 9.3 Циклические зависимости

**Определение:** Модуль A зависит от модуля B, а модуль B зависит от модуля A.

**Решение:**

- Выделить общую функциональность в третий модуль
- Использовать события для развязки
- Пересмотреть границы модулей

---

## 10. Процесс разработки

### 10.1 Добавление новой функциональности

**Обязательные этапы:**

1. Определить предметную область (к какому модулю относится)
2. Если модуль не существует — создать по шаблону
3. Разделить логику по слоям (Routes, Controller, Service, Repository)
4. Добавить валидацию входных данных
5. Добавить mapper для преобразования данных
6. Проверить корректность зависимостей
7. Добавить unit-тесты для Service
8. Добавить integration-тесты для API
9. Обновить документацию

---

### 10.2 Чек-лист архитектурного ревью

**Ревьюер MUST проверить:**

- [ ] Логика находится в правильном слое
- [ ] Нет обратных зависимостей
- [ ] Модуль не превышает допустимое количество зависимостей
- [ ] Публичный API модуля экспортируется через index.js
- [ ] Отсутствует доступ к приватным деталям других модулей
- [ ] Service не знает о req/res
- [ ] Repository не содержит бизнес-логику
- [ ] Controller не содержит SQL запросы
- [ ] Нет дублирования логики из других модулей
