# 🌐 Регламент: API Guidelines & REST Standards

## 0. Область применения

Настоящий регламент устанавливает обязательные требования к проектированию и реализации REST API для всех Backend сервисов.

---

## 1. REST API: Основные требования

### 1.1 Структура URL

**Обязательный формат:**

```
/api/{version}/{resource}
/api/{version}/{resource}/{id}
/api/{version}/{resource}/{id}/{sub-resource}
```

**Требования к ресурсам:**

- Существительное во множественном числе
- Формат kebab-case для составных слов
- Только латинские символы

**FORBIDDEN:**

- Глаголы в URL (/getOrders, /createUser)
- CamelCase или snake_case
- Русские буквы или транслит

---

### 1.2 HTTP методы

**Обязательное соответствие:**

| HTTP метод | Назначение           | Идемпотентность | Тело запроса |
| ---------- | -------------------- | --------------- | ------------ |
| GET        | Получение данных     | Да              | Нет          |
| POST       | Создание ресурса     | Нет             | Да           |
| PUT        | Полная замена        | Да              | Да           |
| PATCH      | Частичное обновление | Нет             | Да           |
| DELETE     | Удаление             | Да              | Нет          |

**Правило идемпотентности:**
Повторный вызов идемпотентного метода с теми же параметрами **MUST** давать тот же результат.

---

### 1.3 Именование ресурсов

**MUST:**

- Множественное число для коллекций
- Существительные, не глаголы
- Kebab-case для составных названий
- Логическая вложенность для связанных ресурсов

**Стандартные шаблоны:**

```
GET    /api/v1/orders              # список
GET    /api/v1/orders/{id}         # один элемент
POST   /api/v1/orders              # создание
PATCH  /api/v1/orders/{id}         # обновление
DELETE /api/v1/orders/{id}         # удаление
GET    /api/v1/orders/{id}/items   # вложенный ресурс
```

## 2. Версионирование API

### 2.1 Стратегия версионирования

**Требование:** API **MUST** включать номер версии в URL.

**Формат:** `/api/v{major}/`

**Правило создания новой версии:**
Новая мажорная версия **MUST** создаваться при:

- Удалении полей из ответа
- Изменении типов данных
- Изменении формата ответа
- Изменении поведения существующих эндпоинтов

**Обратная совместимость (в рамках версии):**
Допускается без создания новой версии:

- Добавление новых полей в ответ
- Добавление новых эндпоинтов
- Добавление опциональных параметров
- Исправление багов

---

### 2.2 Поддержка версий

**Требования:**

- Старая версия **MUST** поддерживаться минимум 6 месяцев после выпуска новой
- Дата окончания поддержки **MUST** быть объявлена заранее (минимум за 3 месяца)
- Документация **MUST** содержать migration guide между версиями

---

## 3. Формат ответа

### 3.1 Единый формат для всего API

**Обязательная структура успешного ответа:**

```json
{
  "success": true,
  "data": { ... }
}
```

**Обязательная структура ответа с ошибкой:**

```json
{
  "success": false,
  "error": {
    "code": "ERROR_CODE",
    "message": "Human-readable message"
  }
}
```

**Требования:**

- Поле `success` **MUST** присутствовать во всех ответах
- При `success: true` поле `data` **MUST** содержать результат
- При `success: false` поле `error` **MUST** содержать код и сообщение
- Формат **MUST** быть единым для всех эндпоинтов

---

### 3.2 Формат списков с пагинацией

**Обязательная структура:**

```json
{
  "success": true,
  "data": {
    "items": [ ... ],
    "pagination": {
      "page": 1,
      "limit": 20,
      "total": 150,
      "totalPages": 8
    }
  }
}
```

**Требования к полям:**

- `items` — массив элементов текущей страницы
- `page` — номер текущей страницы (начинается с 1)
- `limit` — количество элементов на странице
- `total` — общее количество элементов
- `totalPages` — общее количество страниц

---

## 4. HTTP статус-коды

### 4.1 Обязательные статус-коды

**Требование:** API **MUST** использовать корректные HTTP статус-коды.

**Стандартные коды успеха:**

- `200 OK` — успешный GET, PATCH, DELETE
- `201 Created` — успешный POST (создание ресурса)
- `204 No Content` — успешная операция без тела ответа

**Стандартные коды ошибок клиента:**

- `400 Bad Request` — невалидные данные в запросе
- `401 Unauthorized` — отсутствует или неверный токен аутентификации
- `403 Forbidden` — недостаточно прав для выполнения операции
- `404 Not Found` — ресурс не найден
- `409 Conflict` — конфликт данных (дубликат уникального поля)
- `422 Unprocessable Entity` — ошибка бизнес-валидации
- `429 Too Many Requests` — превышен лимит запросов

**Стандартные коды ошибок сервера:**

- `500 Internal Server Error` — внутренняя ошибка сервера
- `503 Service Unavailable` — сервис временно недоступен

---

### 4.2 Правила применения статус-кодов

**Валидация данных (400 vs 422):**

- `400` — структурная валидация (неверный тип, отсутствие обязательного поля, неверный формат)
- `422` — бизнес-валидация (заказ не может быть отменён через 2 часа)

**Аутентификация и авторизация (401 vs 403):**

- `401` — пользователь не аутентифицирован (нет токена или токен недействителен)
- `403` — пользователь аутентифицирован, но не имеет прав на операцию

---

## 5. Пагинация

### 5.1 Query параметры

**Обязательные параметры:**

- `page` — номер страницы (начинается с 1, по умолчанию 1)
- `limit` — количество элементов на странице (по умолчанию 20, максимум 100)

**Правила:**

- Значение `limit` **MUST** быть ограничено максимумом (100)
- При `page=0` или отсутствии параметра использовать `page=1`
- При превышении `totalPages` возвращать пустой массив

**Формат запроса:**

```
GET /api/v1/orders?page=2&limit=50
```

---

### 5.2 Реализация пагинации

**Требования к реализации:**

- Использовать OFFSET и LIMIT в SQL
- Всегда возвращать объект `pagination` даже для пустых результатов
- Кэшировать `total` для больших таблиц

**Обработка граничных случаев:**

- Запрос страницы за пределами диапазона → вернуть пустой массив
- Отрицательные значения `page` или `limit` → вернуть 400
- `limit` больше максимума → использовать максимальное значение

---

## 6. Фильтрация и сортировка

### 6.1 Фильтрация через query параметры

**Формат:**

```
GET /api/v1/orders?status=pending&minTotal=1000&userId=123
```

**Требования:**

- Названия параметров **MUST** соответствовать полям ресурса
- Использовать camelCase для составных названий
- Для диапазонов использовать префиксы `min` и `max`

**Специальные префиксы:**

- `min{Field}` — больше или равно
- `max{Field}` — меньше или равно
- `{field}From` — от даты
- `{field}To` — до даты

---

### 6.2 Сортировка

**Обязательные параметры:**

- `sortBy` — поле для сортировки
- `sortOrder` — направление (`asc` или `desc`)

**Формат запроса:**

```
GET /api/v1/orders?sortBy=createdAt&sortOrder=desc
```

**Требования к безопасности:**

- Использовать whitelist разрешённых полей для сортировки
- Валидировать `sortOrder` (только `asc` или `desc`)
- При неверном `sortBy` возвращать 400 с перечнем допустимых полей

---

## 7. Обработка ошибок

### 7.1 Типизация ошибок

**Требование:** Приложение **MUST** использовать специализированные классы ошибок.

**Обязательные классы:**

- `ValidationError` — ошибки валидации входных данных (400)
- `UnauthorizedError` — не авторизован (401)
- `ForbiddenError` — недостаточно прав (403)
- `NotFoundError` — ресурс не найден (404)
- `ConflictError` — конфликт данных (409)
- `InternalError` — внутренняя ошибка (500)

**Обязательные поля класса ошибки:**

- `name` — название класса
- `message` — человекочитаемое описание
- `code` — машиночитаемый код (UPPER_SNAKE_CASE)
- `statusCode` — соответствующий HTTP статус-код

---

### 7.2 Коды ошибок

**Формат:** UPPER_SNAKE_CASE

**Стандартные коды:**

- `VALIDATION_ERROR` — ошибка валидации
- `NOT_FOUND` — ресурс не найден
- `UNAUTHORIZED` — не авторизован
- `FORBIDDEN` — недостаточно прав
- `CONFLICT` — конфликт (дубликат)
- `INTERNAL_ERROR` — внутренняя ошибка

**Правила именования:**

- Код **MUST** быть уникальным в рамках API
- Код **MUST** быть стабильным (не изменяться)
- Для специфичных ошибок использовать префикс модуля: `ORDER_CANNOT_BE_CANCELLED`

---

### 7.3 Глобальный Error Handler

**Требование:** Приложение **MUST** иметь централизованный обработчик ошибок.

**Функции обработчика:**

- Логирование всех ошибок с контекстом
- Преобразование ошибок в единый формат ответа
- Установка корректного HTTP статус-кода
- Скрытие внутренних деталей в production

**FORBIDDEN в production:**

- Отправка stack trace клиенту
- Раскрытие путей к файлам сервера
- Детали подключения к БД
- Внутренние коды ошибок БД

---

## 8. Аутентификация и авторизация

### 8.1 JWT Tokens

**Требование:** Аутентификация **MUST** использовать JWT токены.

**Обязательные типы токенов:**

- **Access Token** — короткоживущий (15 минут), используется для доступа к API
- **Refresh Token** — долгоживущий (30 дней), используется для обновления access token

**Правила:**

- Access token передаётся в заголовке `Authorization: Bearer {token}`
- Refresh token хранится на клиенте и отправляется только в эндпоинт refresh
- При истечении access token клиент **MUST** запросить новый через refresh token

---

### 8.2 Эндпоинты аутентификации

**Обязательные эндпоинты:**

**Регистрация:**

```
POST /api/v1/auth/register
Request: { email, password, name }
Response: 201 { user, accessToken, refreshToken }
```

**Вход:**

```
POST /api/v1/auth/login
Request: { email, password }
Response: 200 { user, accessToken, refreshToken }
```

**Обновление токена:**

```
POST /api/v1/auth/refresh
Request: { refreshToken }
Response: 200 { accessToken }
```

**Выход:**

```
POST /api/v1/auth/logout
Headers: Authorization: Bearer {access-token}
Response: 204 No Content
```

---

### 8.3 Middleware аутентификации

**Требование:** Проверка токена **MUST** быть выполнена через middleware.

**Обязательные действия middleware:**

- Извлечение токена из заголовка Authorization
- Проверка формата (Bearer {token})
- Верификация JWT подписи
- Проверка срока действия
- Добавление данных пользователя в `req.user`
- Обработка ошибок (401 для невалидного/истёкшего токена)

---

### 8.4 Проверка ролей

**Требование:** Проверка прав доступа **MUST** быть отдельным middleware после аутентификации.

**Правило композиции:**

```
router.delete(
  '/api/v1/orders/:id',
  authenticate,           // проверка токена
  checkRole('admin'),     // проверка роли
  controller.delete
);
```

**Обязательное поведение:**

- Если пользователь не аутентифицирован → 401
- Если роль не подходит → 403

---

## 9. Логирование API

### 9.1 Что логировать

**MUST логировать:**

- Все входящие HTTP запросы (метод, путь, IP, user-agent)
- Все ошибки с полным контекстом и stack trace
- Успешную и неуспешную аутентификацию
- Критичные бизнес-операции (создание заказа, платёж)

**FORBIDDEN логировать:**

- Пароли
- Токены (access/refresh)
- Номера карт
- API ключи
- Полные тела запросов (могут содержать чувствительные данные)

---

### 9.2 Формат логирования запросов

**Обязательные поля лога:**

- Временная метка
- HTTP метод
- URL путь
- HTTP статус-код ответа
- Длительность обработки (в мс)
- IP адрес клиента
- User-Agent
- User ID (если аутентифицирован)

**Требование:** Логирование **MUST** быть выполнено через middleware.

---

### 9.3 Уровни логирования событий

**Правила применения:**

- `error` — все HTTP 5xx, ошибки БД, исключения
- `warn` — HTTP 4xx (кроме 401/404), медленные запросы (> 3 сек)
- `info` — успешные HTTP запросы, бизнес-события
- `debug` — детальная отладочная информация (только dev)

---

## 10. Rate Limiting

### 10.1 Требования к защите

**Обязательное применение:**
Все публичные эндпоинты **MUST** иметь ограничение частоты запросов.

**Стандартные лимиты:**

- Общие эндпоинты: 100 запросов за 15 минут с одного IP
- Аутентификация (login): 5 попыток за 15 минут с одного IP
- Создание ресурсов (POST): 20 запросов за 1 минуту

---

### 10.2 Формат ответа при превышении

**HTTP статус:** 429 Too Many Requests

**Обязательные заголовки ответа:**

```
X-RateLimit-Limit: 100
X-RateLimit-Remaining: 0
X-RateLimit-Reset: 1640000000
Retry-After: 900
```

**Тело ответа:**

```json
{
  "success": false,
  "error": {
    "code": "TOO_MANY_REQUESTS",
    "message": "Rate limit exceeded. Try again in 15 minutes."
  }
}
```

## 11. CORS

### 11.1 Настройка CORS

**Требование:** CORS **MUST** быть настроен с явным whitelist разрешённых источников.

**FORBIDDEN:**

- `Access-Control-Allow-Origin: *` в production
- Динамическое разрешение любых origin без проверки

**Обязательная конфигурация:**

- Whitelist разрешённых доменов (через переменные окружения)
- `credentials: true` (для отправки cookies)
- Явный список разрешённых методов
- Явный список разрешённых заголовков

---

### 11.2 Проверка origin

**Правило:** Origin **MUST** проверяться на соответствие whitelist.

**Действия при несоответствии:**

- Вернуть ошибку CORS
- Залогировать попытку доступа с неразрешённого origin
- Не отправлять CORS заголовки в ответе

---

## 12. Документация API

### 12.1 Требования к документации

**Обязательные элементы документации:**

- OpenAPI 3.0 спецификация
- Описание всех эндпоинтов
- Схемы запросов и ответов
- Примеры запросов
- Описание кодов ошибок
- Руководство по аутентификации

**Формат:** YAML или JSON (OpenAPI/Swagger)

**Доступность:**

- Документация **MUST** быть доступна через `/api/docs`
- Swagger UI **SHOULD** быть доступен в dev/staging

---

### 12.2 Актуальность документации

**Требование:** Документация **MUST** обновляться одновременно с изменением API.

**Проверка актуальности:**

- Автоматическая генерация документации из кода (предпочтительно)
- Ручная проверка при code review
- Блокировка PR при расхождении API и документации

---

## 13. Чек-лист качественного API

**API считается качественным если:**

**Структура и соглашения:**

- [ ] Используется версионирование (/api/v1/)
- [ ] Все URL следуют REST соглашениям
- [ ] Единый формат ответа для всех эндпоинтов
- [ ] Правильные HTTP статус-коды
- [ ] Правильные HTTP методы

**Безопасность:**

- [ ] Аутентификация через JWT
- [ ] Проверка прав доступа (авторизация)
- [ ] Rate limiting настроен
- [ ] CORS правильно сконфигурирован
- [ ] Чувствительные данные не логируются

**Функциональность:**

- [ ] Пагинация для всех списков
- [ ] Фильтрация и сортировка где нужно
- [ ] Централизованная обработка ошибок
- [ ] Валидация всех входных данных

**Мониторинг и поддержка:**

- [ ] Логирование всех запросов
- [ ] Логирование всех ошибок с контекстом
- [ ] Актуальная документация (OpenAPI)
- [ ] Миграционные руководства между версиями

---

## 14. Критерии блокировки PR

**PR MUST быть заблокирован если:**

- Отсутствует версионирование в URL
- Неправильные HTTP статус-коды
- Формат ответа отличается от стандарта
- Отсутствует валидация входных данных
- Отсутствует обработка ошибок
- Чувствительные данные логируются
- CORS настроен с `*` в production
- Отсутствует rate limiting
- Документация не обновлена
