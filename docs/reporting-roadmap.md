# Отчет и дорожная карта по развитию отчетности

Дата: 16.04.2026

## 1. Основание анализа

Анализ выполнен по четырем источникам:

- документу со спецификацией отчетов;
- текущей архитектуре и коду проекта;
- документации iikoTransport API;
- документации iikoServer API.

Дополнительно актуальный реестр источников и сопоставление каталога отчетов с текущими экранами:

- `docs/report-source-registry.md`

## 2. Границы задачи

В итоговый контур развития включены:

- доставка и SLA;
- продажи и меню;
- себестоимость и ассортимент;
- прогноз загрузки производства.

Из контура сознательно исключены:

- ФОТ;
- персонал;
- банкеты и резервы.

## 3. Ключевой вывод

Проект уже имеет сильную базу для аналитики:

- на backend есть готовые доменные модули по выручке, фудкосту, топ-блюдам, стоп-листам, дашборду и операционным метрикам;
- на frontend уже есть экраны для дашборда, выручки, топ-блюд, фудкоста, стоп-листа, клиентов и планов;
- есть общий OLAP-клиент, повторные попытки, деградация при ошибках и повторно используемые графики.

Это означает, что основная доработка — не создание системы с нуля, а расширение текущей аналитической платформы.

## 4. Подтвержденные источники данных

### 4.1 Что брать из iikoTransport API

Использовать для отчетов, где важны статусы доставки и оперативные события:

- сводка доставки за период;
- контроль времени по этапам;
- опоздания доставок;
- KPI курьеров;
- маркетинговые источники;
- меню и стоп-листы;
- предзаказы для прогноза загрузки.

Практически важные методы:

- получение токена доступа;
- список организаций;
- список заказов по статусам и датам;
- список заказов по ревизии;
- маркетинговые источники;
- стоп-листы;
- активные координаты курьеров;
- webhooks по доставкам и стоп-листам.

### 4.2 Что брать из iikoServer API

Использовать для исторической и финансовой аналитики:

- OLAP SALES для выручки, почасовых продаж, меню-микса, скидок;
- OLAP columns для обязательной проверки имен полей перед внедрением;
- складские и балансовые отчеты для точного фудкоста и расхода.

Важно: для каждого нового OLAP-отчета сначала проверять реальные поля через endpoint колонок, а не полагаться только на спецификацию.

## 5. Текущее состояние проекта

### Уже реализовано или близко к готовности

1. Динамика продаж
- реализована;
- есть backend-модуль выручки и отдельный экран.

2. Продажи по группам блюд
- реализованы;
- есть экран топ-блюд и соответствующий API.

3. Себестоимость и расход
- реализованы в рабочем виде;
- есть защита от нестабильного OLAP и деградированный режим.

4. Стоп-листы и доступность позиций
- реализованы частично, но уже полезны для операционной работы;
- есть live-обновление через SSE.

5. Операционные метрики доставки
- реализованы частично;
- уже есть заготовки по среднему времени доставки, приготовления и маршрутам курьеров.

### Не хватает до полного контура

1. Отдельного блока доставки и SLA на уровне UI.
2. Почасового отчета по продажам.
3. Полного отчета по опозданиям и KPI курьеров.
4. Отдельного экрана по маркетинговым источникам.
5. Отчета по акциям и промокодам как самостоятельного сценария.
6. Прогноза загрузки производства.

## 7. Архитектурные выводы по проекту

### Сильные стороны

- модульная структура backend подходит для поэтапного расширения;
- frontend уже собран как аналитическая панель с общими фильтрами, графиками и стором;
- повторно использовать можно текущие подходы из модулей выручки, отчетов и фудкоста.

### Ограничения, которые нужно учитывать

- OLAP может отвечать медленно или нестабильно;
- нельзя маскировать ошибки внешнего iiko пустыми ответами;
- фильтрация по подразделению должна идти по корректным идентификаторам подразделений;
- для операционных экранов потребуется кэш и мягкая деградация, а не жесткие ошибки.

## 8. Рекомендуемая дорожная карта

### Этап 1. Быстрый бизнес-эффект

Срок: 1 спринт

Цель — быстро закрыть самые востребованные пробелы на уже готовой архитектуре.

Состав:

1. Продажи по часам
- добавить backend endpoint с группировкой по часу и дню недели;
- вывести heatmap и bar-chart на фронтенде.

2. Контроль времени доставки
- расширить текущие операционные метрики;
- добавить нарушения SLA, разрез по часам и подразделениям.

3. KPI курьеров
- вынести маршруты и эффективность курьеров в отдельный экран;
- показать количество заказов, среднее время и долю опозданий.

4. Маркетинговые источники
- выделить существующую аналитику каналов в отдельный полноценный отчет;
- добавить динамику по дням и структуру заказов.

### Этап 2. Углубление блока доставки

Срок: 1 спринт

1. Сводка доставки за период
- количество заказов;
- выручка;
- статусы;
- каналы;
- разрез по точкам.

2. Контроль опозданий
- сравнение обещанного и фактического времени;
- анализ по курьерам, зонам и часам.

3. Карта курьеров и заказов
- сначала в режиме polling;
- затем при необходимости перевести на webhook плюс частое обновление.

### Этап 3. Коммерческий и ассортиментный слой

Срок: 1 спринт

1. Акции и промокоды
- отдельный отчет по скидкам;
- топ акций;
- доля скидки в выручке.

2. Меню и ассортимент
- расширить текущий блок стоп-листа до полного справочника доступности и категорий.

### Этап 4. Прогноз загрузки производства

Срок: 1 спринт

1. Историческая модель
- средняя нагрузка по аналогичным дням недели и часам;
- сезонные поправки по праздничным датам.

2. Коррекция предзаказами
- подтягивать подтвержденные предзаказы из iikoTransport;
- накладывать их поверх исторической модели.

3. Визуализация
- почасовой прогноз на день;
- сравнение с прошлой неделей;
- индикатор перегруза по подразделению.

- отдельный раздел прогноза производства.

## 10. Критерии готовности

Отчет считается внедренным только если:

- работает фильтрация по периоду и подразделению;
- данные подтверждаются контрольной сверкой с iiko;
- при проблемах внешнего API экран не падает, а показывает понятное предупреждение;
- есть таблица и визуальная часть, а не только сырые цифры.

## 11. Итог

На текущем этапе проект уже покрывает ядро управленческой аналитики и готов к быстрому расширению.

Рекомендуемый порядок внедрения:

1. продажи по часам;
2. SLA доставки;
3. KPI курьеров и опоздания;
4. маркетинговые источники;
5. акции и промокоды;
6. прогноз загрузки производства.

Такой порядок даст максимальную практическую пользу без лишнего усложнения архитектуры.
