# UI/UX AUDIT

## ЭКСПЕРТНЫЙ UX/UI АУДИТ: PANDA DASHBOARD

**Тип продукта:** B2B Analytics Dashboard (SaaS)  
**Стек:** Vue 3 + shadcn-vue + Tailwind CSS + Pinia  
**Дата аудита:** 18 апреля 2026  
**Аудитор:** GitHub Copilot (Senior Product Design Auditor mode)

---

### 1. EXECUTIVE SUMMARY

Panda Dashboard — это внутренний аналитический инструмент для управления ресторанной сетью. Продукт охватывает 17 экранов: от сводного дашборда до KPI курьеров, стоп-листов, клиентской аналитики и финансового планирования.

**Общее впечатление:** Проект демонстрирует высокий технический уровень реализации и хорошее структурное мышление. Дизайн-система построена правильно, компоненты переиспользуются, состояния (загрузка/ошибка/пусто) реализованы аккуратно. Однако продукт страдает от двух системных UX-проблем: **чрезмерной технической прозрачности** (инженерные метаданные прорываются в пользовательский интерфейс) и **непоследовательности в структуре страниц** (часть экранов живет по одним правилам, часть — по другим).

**Уровень зрелости дизайна:** Высокий технический фундамент, средний продуктовый UX.

### 2. ОБЩИЙ АНАЛИЗ ПРОЕКТА

**Что это:** Внутренний аналитический дашборд ресторанной сети. Не публичный продукт — инструмент для управляющих, операционных менеджеров и аналитиков.

**Для кого:** Предположительно (гипотеза) — руководство и операционный менеджмент сети, маркетологи, финансисты. Аудитория технически грамотная, но не обязательно IT-специалисты.

**Задача продукта:** Мониторинг KPI в реальном времени, анализ продаж, управление доставкой, клиентская аналитика, финансовое планирование.

**Считывается ли ценность из интерфейса:** Частично. Сводный дашборд хорошо агрегирует ключевые метрики, но ценность отдельных разделов (например, "Карта курьеров", "Production Forecast") неочевидна без контекста.

**Проблема понятности:** Система экспонирует слишком много технических концепций — Tier 1/2/3, source: iiko-olap, coverage, last reviewed. Это уместно для разработчиков, но создает когнитивный шум для бизнес-пользователей.

### 3. АНАЛИЗ ДИЗАЙНА

#### Визуальный стиль
**Хорошо:** Проект использует shadcn-vue — современный, минималистичный design system с хорошими дефолтами. Светлая тема выглядит чисто и профессионально.

**Плохо:** Цветовая схема почти полностью монохроматическая (primary — фактически чёрный `0 0% 9%`). Для дашборда, где данные должны "говорить" через визуализацию, это создает эмоционально нейтральный, почти стерильный интерфейс. Нет фирменного акцентного цвета, который бы создавал идентичность продукта. "Panda Dashboard" не имеет никакой цветовой идентичности, кроме иконки-панды.

**Почему это важно:** B2B-дашборды конкурируют не только по функциям, но и по тому, насколько уверенно и современно они выглядят. Монохромный минимализм без выраженного бренда делает продукт "generic" — его легко спутать с любым другим internal tool.

#### Цветовая система
**Хорошо:** CSS-переменные корректно определены для light/dark тем. Chart colors (5 переменных) дают достаточно разнообразия для графиков. Семантические цвета (`success`, `warning`, `destructive`) правильно используются в Badge.

**Плохо:** `--success` и `--warning` не определены в `:root` — они используются в Badge (`bg-success/15 text-success`), но их нет в CSS. Это потенциальный источник визуальных сбоев. Также `--accent` и `--secondary` идентичны — `0 0% 96.1%` в светлой теме, что делает их взаимозаменяемыми и бессмысленными.

#### Типографика
**Хорошо:** Montserrat — правильный выбор для аналитического UI. Иерархия шрифтовых размеров выстроена: `text-2xl` для заголовков страниц, `text-lg` для секций, `text-sm` для данных в карточках. Использование `tabular-nums` в метриках — грамотное решение для предотвращения "прыжков" цифр.

**Плохо:** Заголовки секций (`h2`) в разных компонентах не унифицированы. В `DashboardView` — `text-sm font-semibold uppercase tracking-wide`, в `RevenueView` — `text-lg font-semibold`. Это ломает внутренний ритм страниц.

#### Визуальная иерархия
**Хорошо:** MetricCard с plan progress bar грамотно структурирован — иконка → название → значение → план. LFLBadge правильно размещен в правом верхнем углу, не конкурируя с основным значением.

**Плохо:** `ReportPageHeader` занимает слишком много вертикального пространства. 4 технических блока (Updated at / Source / Coverage / Last reviewed) отображаются всегда, занимая ≈80–100px выше лингии фолда. Это ощутимо толкает контент вниз на каждой странице.

#### Консистентность
**Критично:** Разные страницы используют разные шаблоны заголовков:
- `TopDishesView` — простой `<h1>` без `ReportPageHeader`
- `PlansView` — своя `<div>` с `<h1>` + описание
- Все остальные — `ReportPageHeader`

Это не просто стилистическая проблема — пользователь не знает, чего ожидать при переходе между разделами.

### 4. АНАЛИЗ UI

#### Кнопки
**Хорошо:** Иерархия variant'ов (`default`, `outline`, `ghost`, `secondary`) используется логично. `Button variant="outline"` для вторичных действий, `default` для первичных.

**Плохо:** На странице `PlansView` кнопка "Очистить форму" меняет свой label в зависимости от состояния ("Новый план" / "Очистить форму"). Это типичная anti-pattern: кнопка с изменяющимся смыслом создает когнитивную нагрузку — пользователь не уверен, что произойдет при нажатии.

#### Формы
**Хорошо:** Формы в `PlansView` хорошо структурированы, лейблы над полями, правильные типы input'ов.

**Плохо:** Нет валидации с пользовательским фидбеком. `<Input v-model="form.targetValue" type="number" min="0">` — браузерная валидация без кастомного сообщения об ошибке. При вводе некорректных данных пользователь не получает объяснения.

**Плохо:** В `ClientsView` форма фильтров переусложнена: в одной карточке совмещены Terminal Group, статусы заказов (toggle-кнопки), флаг включения профиля, режим enrichment и лимит профилей. Это 5 параметров с нелинейными зависимостями — слишком высокая когнитивная нагрузка для фильтровой панели.

#### Карточки
**Хорошо:** `MetricCard` — одно из лучших решений в проекте. Все состояния (loading skeleton, error, normal) реализованы корректно. Прогресс-бар плана с цветовым семафором работает как быстрый визуальный сигнал.

**Плохо:** В `StopListView` используется другой шаблон карточек — `Card + CardHeader + CardContent` без унифицированного padding. Выглядит немного иначе, чем MetricCard, хотя несёт ту же смысловую нагрузку (одно значение + подпись).

#### Навигация (Sidebar)
**Хорошо:** Сайдбар коллапсируется (широкий/иконки), есть мобильный drawer с backdrop, активный пункт выделен акцентным цветом, groups разбиты по доменам.

**Плохо — критично:** Статусные бейджи в навигации (`Ready`, `Beta`, `Partial`, `Planned`) написаны на **английском**, тогда как весь интерфейс на **русском**. Это грубое нарушение языкового единообразия и создает ощущение "недоделанного" продукта.

**Плохо:** Бейдж `Planned` у недоступных разделов создает UX-ловушку: пользователь кликает, переходит на страницу, которая либо пустая, либо не работает. Нет никакой защиты от этого сценария (нет disabled состояния, нет placeholder-страницы).

**Плохо:** Иерархия разделов в меню требует от пользователя знания терминологии ("SLA доставки", "Опоздания", "KPI курьеров", "Карта курьеров" — это 4 пункта в одном блоке "Операции"). Для нового пользователя непонятно, чем они отличаются.

#### Таблицы
**Хорошо:** В `TopDishesView` реализована сортировка колонок с иконкой направления, поиск в реальном времени, разделение на "Топ" и "Аутсайдеры".

**Плохо:** `min-w-[980px]` у таблицы — это жёсткий минимум, что делает таблицу горизонтально скроллируемой на большинстве планшетов. При этом скроллбар не стилизован и не имеет visual hint о существовании прокрутки.

#### Состояния элементов
**Хорошо:** Загрузочные skeleton'ы правильно воспроизводят структуру будущего контента (пустые MetricCard с pulse анимацией), а не просто показывают spinner.

**Плохо:** Пустое состояние ("Выберите организацию и период") с иконкой и одной строкой текста — слишком минималистично. Нет подсказки, **что именно нужно сделать** и **где** это сделать. Иконка `BarChart2` никак не связана с действием.

### 5. АНАЛИЗ UX

#### Понятность первого экрана
**Дашборд** (`/dashboard`) — это правильная точка входа. KPI сразу видны, структура последовательная. Однако под KPI-картами идет огромный `ReportPageHeader` с технической информацией — "Updated at", "Source: iiko-cloud + iiko-olap", "Coverage", "Last reviewed". Для менеджера, пришедшего посмотреть выручку за день, это мусор.

#### Логика пользовательского пути
**Гипотеза (нет данных о реальных сессиях):** Типичный сценарий: войти → проверить сводку → перейти в нужный раздел → применить фильтры → прочитать данные. Этот путь технически работает, но перегружен промежуточными решениями:
1. При переходе в любой раздел нужно применить фильтры вручную
2. Нет "умного" восстановления последнего использованного фильтра при возврате
3. Нет breadcrumbs — непонятно, где ты находишься в иерархии

#### Фильтровая система
**Проблема:** `PageFilters` — это унифицированный компонент, но его логика не всегда очевидна:
- На дашборде (`mode="date"`) выбирается одна дата
- В остальных разделах (`mode="range"`) — период
- Есть LFL-расчет, но нет объяснения **что такое LFL** прямо рядом с фильтром

Пользователь из маркетинга или операций не обязан знать, что LFL = Like-for-Like, это маркетинговый термин для сравнения сопоставимых периодов.

#### Критические UX-проблемы

**🔴 КРИТИЧНО: Инженерные метаданные в пользовательском интерфейсе**  
`ReportPageHeader` показывает: `Tier 1`, `Source: iiko-olap`, `Coverage`, `Last reviewed: 18.04.2026`. Это инструменты для DevOps-мониторинга, не для бизнес-пользователей. Руководитель сети не знает, что такое "Tier 1" или "iiko-olap". Эти данные нужно убрать из основного UI или скрыть за иконкой ℹ️.

**🔴 КРИТИЧНО: Английский язык в навигации**  
`Ready`, `Beta`, `Partial`, `Planned` — в русскоязычном приложении. Смешение языков — это не просто эстетика, это сигнал незавершённости, который подрывает доверие к продукту.

**🔴 КРИТИЧНО: Переход на "planned" разделы**  
Пользователь может кликнуть на "Карта курьеров" (Planned), перейти на страницу и увидеть либо пустой контент, либо partial-функциональность без объяснений. Нет ни блокировки, ни информативного placeholder'а.

#### Средние UX-проблемы

**🟡 СРЕДНЕ: Отсутствие глобального контекста фильтра**  
Каждый раздел управляет своим фильтром независимо. При переходе из "Выручки" (где выбран период 1–15 апреля) в "Топ блюд" — пользователь видит пустой экран и должен заново применять фильтры. Нет персистентности фильтра между разделами одного домена.

**🟡 СРЕДНЕ: Перегруженная фильтровая карточка в ClientsView**  
5 параметров с нелинейной зависимостью в одной карточке — Terminal Group + Статусы + Профиль + Режим + Лимит. Пользователь не понимает, как параметры влияют друг на друга.

**🟡 СРЕДНЕ: Отсутствие drill-down**  
MetricCard "Заказов: 1 432" — это конечная точка. Нет возможности кликнуть и увидеть детализацию. В аналитических инструментах drill-down — стандартное ожидание пользователя.

#### Мелкие UX-недостатки

**🟢 МЕЛКО:** Кнопка "Обновить" в `ReportPageHeader` показывает "Обновление..." при загрузке, но не возвращается в нормальное состояние с явным подтверждением успеха.

**🟢 МЕЛКО:** Пустое состояние с `BarChart2` иконкой и текстом "Выберите организацию и период" не указывает, где именно находятся эти контролы.

**🟢 МЕЛКО:** В `TopDishesView` заголовок страницы — просто `<h1>Топ блюд</h1>` без бейджей статуса и readiness. Пользователь не знает, насколько данные надёжны.

### 6. СИЛЬНЫЕ СТОРОНЫ ПРОЕКТА

**1. Архитектура компонентов** — `MetricCard` покрывает все необходимые состояния (loading/error/data/plan/lfl). Это зрелое решение, которое уменьшает количество багов в UI и ускоряет разработку новых экранов.

**2. Поддержка тёмной темы** — реализована через CSS-переменные корректно. Переключение работает, цветовые роли сохраняются в обоих режимах.

**3. Loading skeleton'ы** — вместо spinners используются skeleton-блоки, которые воспроизводят структуру контента. Это снижает воспринимаемое время загрузки и предотвращает layout shift.

**4. LFLBadge** — элегантное решение для отображения динамики: направление (↑/↓), цвет (green/red), инверсия для "плохих" метрик (время, дисконт). Правильная бизнес-логика, правильная реализация.

**5. Plan progress bar в MetricCard** — визуальный семафор выполнения плана прямо в KPI-карточке. Это информация высокой плотности: текущее / плановое / процент — всё в одной карточке без перехода на отдельный экран.

**6. Структура навигации по доменам** — разбивка на "Обзор / Операции / Продажи / Клиенты / Финансы" логично отражает оргструктуру ресторанного бизнеса.

**7. Адаптивный сайдбар** — desktop коллапс + mobile drawer — правильная реализация для гибридного использования на десктопе и планшете.

**8. Матрица зрелости функций** — наличие `featureReadiness.js` и матрицы — редкость для internal tools. Это позволяет честно коммуницировать статус функций. Задумка правильная, но реализация в UI проблемная.

### 7. СЛАБЫЕ СТОРОНЫ ПРОЕКТА

**1. Технические метаданные в пользовательском интерфейсе** — `Tier 1`, `Source: iiko-olap`, `Coverage`, `Last reviewed` — занимают видное место на каждой странице. Критичность: HIGH.

**2. Языковое смешение** — `Ready/Beta/Partial/Planned` в русскоязычном продукте. Критичность: HIGH.

**3. Отсутствие персистентности фильтров** — Каждый переход между разделами сбрасывает контекст. Критичность: MEDIUM.

**4. Непоследовательность шаблонов страниц** — `TopDishesView` живет по другим правилам, чем остальные. Критичность: MEDIUM.

**5. Отсутствие drill-down на KPI** — Нет возможности кликнуть на метрику и перейти к детализации. Критичность: HIGH.

**6. Перегруженная фильтровая панель на ClientsView** — Слишком много параметров с нелинейной зависимостью. Критичность: MEDIUM.

**7. Навигационная ловушка с planned-разделами** — Пользователь попадает на нерабочие страницы без объяснений. Критичность: HIGH.

**8. Отсутствие брендового цвета** — Монохромная система с чёрным primary. Критичность: LOW.

### 8. ПЛЮСЫ И МИНУСЫ

#### Плюсы
- Зрелый компонентный фундамент (shadcn-vue + MetricCard + LFLBadge)
- Все критические состояния UI реализованы (loading/error/empty)
- Тёмная тема работает корректно
- Структурированная навигация по бизнес-доменам
- Plan tracking прямо в KPI-карточках
- Адаптивность (desktop + mobile drawer)

#### Минусы
- Инженерные метаданные загрязняют бизнес-UI
- Смешение языков (RU + EN) в одном интерфейсе
- Пользователь попадает на неработающие страницы (planned-разделы)
- Нет персистентности фильтров между разделами
- Нет drill-down — данные "мертвы", на них нельзя кликнуть
- Непоследовательные шаблоны страниц
- Монохромная идентичность без брендового акцента

# FUNCTIONAL & LOGIC AUDIT

## ФУНКЦИОНАЛЬНЫЙ И ЛОГИЧЕСКИЙ АУДИТ: PANDA DASHBOARD

**Тип продукта:** B2B Analytics Dashboard  
**Стек бэкенда:** Node.js + Express, файловое хранилище (JSON), iiko Server API / iiko OLAP API  
**Стек фронтенда:** Vue 3 + Pinia + Axios  
**Дата аудита:** 18 апреля 2026

---

### 1. EXECUTIVE SUMMARY

Panda Dashboard — многомодульная аналитическая система, которая агрегирует данные из двух внешних источников (iiko Server API и iiko OLAP) и предоставляет бизнес-пользователям отчёты в режиме, близком к реальному времени.

**Как работает продукт в целом:** Фронтенд (Vue 3 SPA) отправляет запросы на собственный бэкенд (Node.js/Express), который в свою очередь запрашивает iiko API, агрегирует данные и возвращает готовый ответ. Единственный модуль с постоянным хранилищем — Plans (данные в `plans.json`).

**Насколько логика целостна:** Логика отдельных модулей реализована аккуратно. Однако система имеет **несколько критических структурных проблем:**
1. Файловое хранилище планов — не потокобезопасное и не устойчивое к сбоям
2. Кэш организаций и выручки хранится в памяти процесса без персистентности — перезапуск сбрасывает всё
3. Таймзонная логика реализована частично: фронтенд работает в локальном времени браузера, бэкенд — в московском, без явного согласования
4. Нет аутентификации пользователей — любой, кто знает URL бэкенда, получает полный доступ ко всем данным
5. Механизм расчёта LFL реализован только на фронтенде (подбор дат), но не верифицируется и не объясняется бэкендом

---

### 2. ОБЩАЯ ЛОГИКА ПРОДУКТА

#### Описание системы

```
[Браузер / Vue SPA]
    ↕ HTTP (axios)
[Node.js Backend / Express]
    ↕ HTTP (axios + cookie-jar)
[iiko Server API]   [iiko OLAP API]
```

**Модули бэкенда:**
- `organizations` — список подразделений, кэш 5 мин в памяти
- `dashboard` — сводные KPI за одну дату, агрегирует `revenue` по всем org
- `revenue` — OLAP-отчёт по выручке, кэш 2 мин в памяти
- `stopList` — список позиций в стоп-листе + расчёт упущенной выгоды, кэш 20–45 сек в памяти
- `topDishes` — топ блюд по OLAP, используется также в stopList для расчёта упущенной выгоды
- `clients` — клиентская аналитика по OLAP
- `clientAnalytics` — расширенный анализ клиентов (когорты и т.д.)
- `foodcost` — фудкост по OLAP
- `plans` — CRUD-операции, хранилище: `data/plans.json`
- `reports` — агрегатор отчётов
- `webhook` — приём SSE-событий от iiko о стоп-листе

#### Основные сценарии
1. Пользователь открывает дашборд → бэкенд запрашивает выручку у всех org параллельно → агрегирует → возвращает KPI
2. Пользователь открывает Выручку → выбирает период и org → бэкенд запрашивает OLAP → возвращает разбивку
3. Пользователь открывает Стоп-лист → бэкенд запрашивает актуальные данные iiko → добавляет расчёт упущенной выгоды
4. Пользователь создаёт/редактирует план → запрос к бэкенду → запись в `plans.json`
5. iiko присылает webhook о стоп-листе → бэкенд транслирует событие всем SSE-клиентам

### 3. USER FLOWS

#### Flow 1: Просмотр дашборда

**Шаг 1:** Пользователь открывает `/dashboard`  
**Шаг 2:** Фронтенд берёт `today` из `FiltersStore` (локальное время браузера)  
**Шаг 3:** `dashboardApi.getDashboard({ date: '2026-04-18' })` → бэкенд  
**Шаг 4:** Бэкенд конвертирует дату в московское время через `toMoscowDateStr`  
**Шаг 5:** Параллельно запрашивает OLAP по каждой org с таймаутом 30 сек  
**Шаг 6:** Агрегирует результаты, возвращает `summary` + `byOrganization` + `revenueByChannel`

**Разрыв 1:** Если пользователь находится в часовом поясе UTC+7 и открывает дашборд в 23:30 местного — фронтенд передаёт "сегодня" (местная дата), а бэкенд переводит в московское время. Это может привести к отображению данных **другой даты** без предупреждения.

**Разрыв 2:** При таймауте одной из org (>30 сек) она попадает в `error: true` в `byOrganization`, но суммарные KPI формируются **без этих данных** без явного предупреждения пользователю. Пользователь видит заниженную выручку и не знает почему.

#### Flow 2: Создание плана

**Шаг 1:** Пользователь заполняет форму (метрика + период + org + значение)  
**Шаг 2:** `plansApi.createPlan(payload)` → POST `/api/plans`  
**Шаг 3:** `PlansService.normalizePlanPayload` валидирует  
**Шаг 4:** Проверка дубликата: одна комбинация `(metric, period, orgId)`  
**Шаг 5:** Генерация ID: `plan_${Date.now()}_${random}`  
**Шаг 6:** `plans.unshift(plan)` → `writeAll` → запись в файл

**Разрыв:** Между чтением файла (шаг 4) и записью (шаг 6) нет блокировки. При **двух одновременных запросах** оба пройдут проверку дубликата успешно — в файл запишутся два одинаковых плана. Race condition.

**Разрыв:** При неудачной записи в файл (`writeFile` throws) ответ клиенту уже не отправлен, но в `plans.value` на фронтенде ничего не изменилось — пользователь увидит ошибку. Однако данные частично могли быть записаны (truncated file).

#### Flow 3: Стоп-лист + упущенная выгода

**Шаг 1:** Запрос стоп-листа  
**Шаг 2:** Параллельно: `topDishesService.getDishesDataset` за последние N дней  
**Шаг 3:** Для каждой позиции в стоп-листе: берём `revenue / hours` из истории продаж → оценка упущенной выгоды  
**Шаг 4:** Если `topDishes` недоступен → `degraded: true`, выручка = 0 для этой org

**Разрыв:** Расчёт упущенной выгоды основан на **средней выручке в час** по всему периоду lookback (14 дней по умолчанию). Это не учитывает сезонность, день недели и время суток. Расчёт корректен математически, но может давать значительно завышенные оценки для ночных стоп-листов.

**Разрыв:** `_deduplicateItems` при совпадении ID оставляет запись с **более поздним** `startedAt`. Это правильно для активных позиций, но если iiko вернул позицию с некорректным timestamp — она "выиграет" у корректной записи без каких-либо проверок.

#### Flow 4: SSE / Real-time стоп-лист

**Шаг 1:** Фронтенд подключается к `/api/stop-lists/events` (EventSource)  
**Шаг 2:** iiko присылает webhook → бэкенд транслирует `stopListUpdate` всем SSE-клиентам  
**Шаг 3:** Фронтенд при получении события обновляет стоп-лист

**Разрыв:** Нет механизма переподключения SSE на фронтенде при разрыве соединения (браузер делает это автоматически, но нет отображения состояния соединения пользователю). Если соединение разорвано, пользователь не видит новые стоп-листы и не знает об этом.

**Разрыв:** Webhook не верифицирует источник (IIKO_WEBHOOK_TOKEN опционален). При `webhookToken = undefined` — любой может прислать `StopListUpdate` и инициировать обновление данных у всех клиентов.

### 4. БИЗНЕС-ЛОГИКА

#### Нормализация каналов продаж

`_normalizeChannelName` в `RevenueService` нормализует названия каналов по ключевым словам:
- "яндекс" → "Яндекс.Еда"
- "самовывоз/самовынос/с собой" → "Самовынос"
- "достав/курьер" → "Доставка"
- всё остальное → "Зал"

**Проблема:** Fallback "Зал" — это слишком широкое обобщение. Если iiko добавит новый тип канала (например, "Собственный агрегатор"), он молча попадёт в "Зал", искажая данные без каких-либо предупреждений.

**Проблема:** "курьер" как ключевое слово для определения "Доставки" конфликтует с "KPI курьеров" в операционной семантике. Если iiko назовёт тип заказа "Курьер (зал)" — это некорректно попадёт в "Доставка".

#### LFL-расчёт

LFL (Like-for-Like) рассчитывается **только на фронтенде** в `usePeriod.js`:
- `getLFLRange` берёт текущий диапазон и сдвигает его назад на год
- Этот диапазон передаётся в запрос как `lflDateFrom` / `lflDateTo`
- Бэкенд делает **два независимых запроса к OLAP** и возвращает оба набора данных
- Процент LFL вычисляется на бэкенде как `(current - previous) / previous * 100`

**Проблема:** Нет проверки, что LFL-период является "сопоставимым" (например, одинаковое количество дней, одинаковые типы дней — рабочие/выходные). "Текущая неделя" → LFL = "та же неделя год назад" — корректно. Но "произвольный период" может не иметь смысла для LFL.

**Проблема:** Расчёт LFL процента: `(current - previous) / previous * 100`. Если `previous = 0` — деление на ноль. Нет защиты.

#### Расчёт дисконта

В `DashboardService`:
```js
const discountSum = Math.max(0, totalRevenueBeforeDiscount - totalRevenue);
const discountPercent = totalRevenueBeforeDiscount > 0 
  ? Math.round((discountSum / totalRevenueBeforeDiscount) * 10000) / 100 
  : 0;
```

Дисконт вычисляется как разница между `revenueBeforeDiscount` и `revenue`. Это корректно при условии, что iiko возвращает `discountSum` правильно. Однако если iiko возвращает `discountSum = 0` при ненулевом дисконте — формула даст нулевой результат без каких-либо предупреждений.

#### Matching планов с KPI

`getMetricPlan` в `PlansStore`:
1. Ищет точный план для `(metric, period, orgId)`
2. Fallback: общий план для `(metric, period, "")` — если конкретная org не найдена

**Проблема:** `period` в планах — строка ("2026-04", "Q1-2026"), а период в фильтрах — диапазон дат. Нет явного маппинга "диапазон дат → строковый период плана". Это значит, что план за "Апрель 2026" не будет автоматически применяться при выборе пользователем "01.04.2026 — 30.04.2026" через произвольный период.

### 5. ВАЛИДАЦИИ И ОГРАНИЧЕНИЯ

#### Бэкенд-валидации

| Модуль | Что валидируется | Что не валидируется |
|--------|-----------------|---------------------|
| Plans | metric, period (непустые строки), targetValue > 0 | Допустимые значения metric и period (любая строка пройдёт) |
| Dashboard | наличие date | Формат date (любая невалидная строка будет передана в OLAP) |
| Revenue | organizationId, dateFrom, dateTo (через `requestValidation`) | Максимальный диапазон дат |
| StopList | organizationIds через `parseIdQuery` | Корректность значений внутри массива |
| Webhook | наличие `webHooksUri` | Формат URI (любая строка пройдёт) |

#### Что не валидируется — критично

**1. `metric` и `period` в планах** — принимаются любые строки. Можно создать план с `metric: "xss_payload"` или `period: "not-a-period"`. Данные сохранятся в файл, план появится в списке, но не будет совпадать ни с одним KPI.

**2. Дата в дашборде** — `new Date(date)` из запроса. Если передать `date: "invalid"` — `day.toISOString()` бросит исключение, которое не перехвачено в `getDashboardData`. Это приведёт к 500 без осмысленного сообщения.

**3. Размер запроса** — нет ограничений на размер тела POST-запроса к `/api/plans`. Можно передать гигабайтный payload.

**4. Тип данных в теле запроса** — `normalizePlanPayload` делает `String(payload.metric || "")`, что защищает от примитивов, но если `metric` — это объект или массив, `String({})` вернёт `"[object Object]"` и пройдёт валидацию.

**5. Отрицательный `targetValue`** — защита есть (`targetValue <= 0`), но `targetValue = -1` не пройдёт. Однако `targetValue = 0.001` пройдёт — план на 0.001 рублей создастся. Это вероятно не является проблемой, но нет нижнего порога разумности.

### 6. СОСТОЯНИЯ И ПЕРЕХОДЫ

#### Состояния DashboardStore

```
[idle] → loadDashboard() → [loading]
  [loading] → success → [data]
  [loading] → error → [error]
  [loading] → abort (новый запрос) → [loading] (новый)
  [data] → reload → [loading]
  [error] → reload → [loading]
```

**Хорошо:** AbortController предотвращает race condition при множественных вызовах `loadDashboard`. `requestId` защищает от обработки устаревших ответов.

**Проблема:** После `reset()` (который вызывается при смене маршрута) `dashboardData = null` и `error = null`. Если пользователь быстро переключается между вкладками, на долю секунды `data = null` и `error = null` одновременно — UI показывает "пустое состояние" с иконкой вместо skeleton. Это мигание контента, которое ухудшает воспринимаемую производительность.

#### Состояния PlansStore

```
[loaded] → createPlan() → [saving]
  [saving] → success → loadPlans() → [loading] → [loaded]
  [saving] → error → [loaded] (plans не изменились)
  
[loaded] → deletePlan(id) → [saving]
  [saving] → success → optimistic delete → [loaded] (локальное удаление)
  [saving] → error → [loaded] (элемент остался)
```

**Проблема:** При `createPlan` — после успешного создания вызывается `loadPlans()`. Это перезагружает **весь список** с сервера. Если список большой, это добавляет задержку. Эффективнее было бы добавить созданный план локально (optimistic update).

**Проблема:** При `deletePlan` — сделан optimistic delete (план удаляется из `plans.value` локально сразу). Если запрос провалится — план возвращается в список? Нет — в блоке `catch` нет восстановления. Пользователь видит, что план исчез, но на сервере он остался. После следующего `loadPlans()` план появится снова — это создаёт непредсказуемое поведение.

#### Состояния фильтров

`FiltersStore` — единственный глобальный store для фильтров. Состояние: `preset`, `customFrom`, `customTo`, `organizationId`.

**Проблема:** `organizationId` в `FiltersStore` дублируется с `currentOrganizationId` в `RevenueStore`. В `PageFilters.vue` используется `revenueStore.currentOrganizationId`, а `filtersStore.organizationId` обновляется через `filtersStore.setOrganization(val)` синхронно в том же computed setter. Два источника правды для одного и того же значения — потенциальный источник рассинхронизации.

### 7. ОШИБКИ И EDGE CASES

#### Как обрабатываются ошибки

**Хорошо:**
- `Promise.allSettled` в `DashboardService` обрабатывает частичные ошибки при параллельных запросах к org
- `withRetry` в `OlapClient` делает до 4 попыток с экспоненциальным backoff для сетевых ошибок
- `withTimeout` в `DashboardService` предотвращает вечное ожидание ответа (30 сек)
- `apiClient` в Axios перехватывает все ошибки и показывает toast-уведомление

**Плохо / не обрабатывается:**

| Сценарий | Что происходит | Последствие |
|----------|----------------|-------------|
| `new Date("invalid")` в `getDashboardData` | Uncaught exception → 500 | Неинформативная ошибка |
| `plans.json` содержит невалидный JSON | `JSON.parse` → catch → возвращает `[]` | Все планы исчезают без предупреждения |
| `plans.json` заблокирован другим процессом | `fs.writeFile` throws | План не сохранён, ошибка 500 |
| IIKO_USER / IIKO_PASSWORD не заданы | Запрос к iiko провалится с 401 | Все отчёты недоступны без внятного объяснения |
| LFL: `previousRevenue = 0` | Деление на ноль → `NaN` или `Infinity` | LFLBadge показывает некорректное значение |
| Стоп-лист: `endTs < startTs` в `getDuration` | Возвращает `null` для всех длительностей | Позиция показывается без времени в стоп-листе |
| SSE: клиент отключается | `sseService` должен убрать его из списка | Если cleanup не реализован → memory leak |
| Dashboard: все org вернули ошибку | `totalRevenue = 0`, `byOrganization` все с `error: true` | Показывается нулевая выручка, не ошибка |

#### Критические edge cases

**🔴 Race condition в plans.json**  
Два одновременных POST `/api/plans` → оба читают файл → оба не видят дубликата → оба записывают → второй перезаписывает первый. Итог: в файле один план, но пользователь создал два.

**🔴 Коррупция plans.json**  
Если процесс падает во время `fs.writeFile` (например, kill -9) — файл может оказаться частично записанным. При следующем чтении `JSON.parse` упадёт → все планы исчезнут. Нет резервных копий, нет atomic write.

**🔴 Dashboard с нулевой выручкой при ошибках API**  
Если все org вернули таймаут, дашборд покажет `totalRevenue = 0`, `totalOrders = 0`. Пользователь видит нулевую выручку вместо сообщения "Данные временно недоступны". Это может быть принято за реальные данные (нерабочий день, закрытие).

**🟡 Устаревший кэш при смене данных в iiko**  
`RevenueService` кэширует отчёты на 2 минуты в памяти. Если в iiko внесли корректировку — пользователь увидит старые данные до истечения TTL без возможности принудительно очистить кэш (кнопка "Обновить" делает новый запрос, но кэш ещё жив).

**🟡 Пересечение concurrent requests**  
`OlapClient.maxConcurrentRequests = 1` — сериализует все OLAP-запросы глобально. При одновременном открытии дашборда несколькими пользователями их запросы встанут в очередь. Это намеренно (защита от перегрузки iiko), но в документации не указано.

### 8. РОЛИ И ДОСТУПЫ

#### Текущая ситуация

**Аутентификации нет.** Любой HTTP-клиент, знающий URL бэкенда, может:
- Читать все аналитические данные (выручку, клиентов, фудкост)
- Создавать, редактировать и удалять планы
- Зарегистрировать webhook на произвольный URL
- Инициировать обновление стоп-листа у всех подключённых SSE-клиентов

**CORS:** Настроен через `CORS_ORIGIN` env. По умолчанию `"*"` — разрешает запросы с любого домена. Это нормально для internal tool без auth, но в сочетании с отсутствием аутентификации создаёт риск CSRF-подобных атак.

**Webhook-токен:** `IIKO_WEBHOOK_TOKEN` опционален. Если не задан — `/webhooks/iiko` принимает запросы от кого угодно.

#### Что это означает на практике

Если URL бэкенда известен третьим лицам — они могут:
1. Получить полную финансовую отчётность сети
2. Испортить все планы (удалить или перезаписать)
3. Вызвать SSE-шторм (флуд webhook-событиями → все клиенты начнут обновлять стоп-лист)

### 9. ДАННЫЕ

#### Как создаются

- Аналитические данные: **только чтение** из iiko API, бэкенд агрегирует и трансформирует
- Планы: создаются пользователем через форму, хранятся в `plans.json`

#### Как хранятся

- Аналитика: **только в памяти** (кэш Map), TTL 2–5 минут
- Планы: `backend/data/plans.json` — плоский JSON-массив

#### Риски хранилища планов

| Риск | Вероятность | Последствие |
|------|-------------|-------------|
| Race condition при одновременных записях | Средняя | Потеря данных |
| Коррупция файла при сбое процесса | Низкая | Потеря всех планов |
| Неограниченный рост файла | Низкая | Замедление чтения |
| Отсутствие резервных копий | Постоянное | Данные невосстановимы после сбоя |

#### Таймзоны

Бэкенд использует московское время (`Europe/Moscow`) через `toMoscowDateStr` при формировании OLAP-запросов. Однако:
- Фронтенд вычисляет "today" в локальном времени браузера
- При передаче даты строкой (`"2026-04-18"`) бэкенд корректно конвертирует её в московское
- Но метаданные ("Updated at" в `ReportPageHeader`) показываются в UTC/локальном времени без явного указания таймзоны

**Сценарий коллизии:** Пользователь во Владивостоке (UTC+10) в 01:00 открывает дашборд. "Сегодня" для него — 19 апреля. Бэкенд переводит в московское — 18 апреля. Пользователь думает, что смотрит данные за сегодня, но видит вчерашние данные по московскому времени. Нет никакого предупреждения.

### 11. РЕКОМЕНДАЦИИ (ФУНКЦИОНАЛЬНЫЕ)

#### 11.1 Добавить базовую аутентификацию
**Что сделать:** API-key в заголовке `X-API-Key`, проверяемый middleware на всех роутах. Для внутреннего инструмента это минимально необходимый уровень защиты.  
**Эффект:** Закрывает несанкционированный доступ к данным и операциям.

#### 11.2 Заменить file-based хранилище планов на atomic write + mutex
**Что сделать:** Использовать `write-file-atomic` (npm) для атомарных записей. Добавить простой mutex (например, через `async-mutex`) для сериализации операций с файлом.  
**Эффект:** Устраняет race condition и риск коррупции файла.

#### 11.3 Dashboard: явная индикация частичных данных
**Что сделать:** Если хотя бы одна org вернула ошибку — добавить в ответ `hasErrors: true` и список проблемных org. Фронтенд должен показывать предупреждение "Данные по X подразделениям недоступны".  
**Эффект:** Пользователь не принимает решения на основе неполных данных.

#### 11.4 Защитить LFL от деления на ноль
**Что сделать:** В расчёте LFL: `if (previous === 0) return null`. LFLBadge при `value = null` не должен отображаться.  
**Эффект:** Устраняет NaN/Infinity в UI.

#### 11.5 Сделать IIKO_WEBHOOK_TOKEN обязательным
**Что сделать:** Если `IIKO_WEBHOOK_TOKEN` не задан — логировать WARN при старте. Опционально — отклонять webhook-запросы без токена.  
**Эффект:** Закрывает возможность внешнего SSE-флуда.

#### 11.6 Устранить два источника правды для organizationId
**Что сделать:** Удалить `currentOrganizationId` из `RevenueStore`. Использовать только `filtersStore.organizationId` как единственный источник.  
**Эффект:** Устраняет потенциальную рассинхронизацию.

#### 11.7 Показывать таймзону в "Updated at"
**Что сделать:** Добавить явное указание часового пояса в метаданные (`Updated at: 18.04.2026 15:30 МСК`).  
**Эффект:** Устраняет путаницу для пользователей в других часовых поясах.

#### 11.8 Rollback при ошибке optimistic delete
**Что сделать:** В `PlansStore.deletePlan` сохранять snapshot перед удалением и восстанавливать его при ошибке.  
**Эффект:** UI остаётся консистентным с сервером.

#### 11.9 Валидация допустимых значений metric и period в планах
**Что сделать:** Передавать на бэкенд enum допустимых значений для `metric` и валидировать формат `period` (например, `/^\d{4}-\d{2}$/` для месячных планов).  
**Эффект:** Исключает мусорные данные в `plans.json`.

#### 11.10 Резервное копирование plans.json
**Что сделать:** Перед каждой перезаписью создавать `plans.json.bak`. При сбое `JSON.parse` — восстанавливать из backup.  
**Эффект:** Устраняет риск полной потери планов при коррупции файла.

### 13. СТРАТЕГИЧЕСКИЕ УЛУЧШЕНИЯ (ФУНКЦИОНАЛЬНЫЕ)

#### 13.1 Перейти на настоящую базу данных для планов
Текущее `plans.json` — временное решение, которое не масштабируется. Даже SQLite (через `better-sqlite3`) даст транзакции, атомарность и надёжность без необходимости разворачивать сервер БД.

#### 13.2 Добавить слой аутентификации
Минимум: JWT-токены с коротким TTL + refresh token. Оптимум: интеграция с существующей системой SSO/аутентификации ресторанной сети.

#### 13.3 Кэширование с возможностью инвалидации
Текущий in-memory кэш сбрасывается при перезапуске. Перенести кэш в Redis (или даже в файлы с TTL) позволит выдержать рестарты и обеспечить инвалидацию по событию (webhook → инвалидировать кэш стоп-листа).

#### 13.4 Добавить идемпотентность в мутирующие операции
Для Plans API: принимать `idempotencyKey` в заголовке или теле запроса. При повторном запросе с тем же ключом — возвращать уже созданный результат. Это защищает от двойного создания плана при сетевой ошибке.

#### 13.5 Проектировать Plans как domain-объекты с explicit периодами
Текущие планы — строковые `period: "2026-04"`. Нет явной связи с диапазонами дат. Перейти на `{ year: 2026, month: 4, type: "monthly" }` с вычисляемыми `dateFrom`/`dateTo`. Это позволит автоматически матчить план с выбранным периодом.

# 1. EXECUTIVE SUMMARY

Модуль «Стоп-лист» — это **операционный дашборд мониторинга реального времени**, встроенный в панель управления ресторанной сетью (`panda-dashboard`). Он подключается к системе iiko (облачная POS-платформа для общепита) и отображает текущий перечень позиций меню, которые помечены как недоступные на кассовых терминалах ресторана (закончились, изъяты, временно выключены).

**Для чего нужен**: дать менеджерам и аналитикам ресторанной сети централизованный, читаемый интерфейс поверх «сырых» данных iiko, обогащённый вычисленной метрикой упущенной выручки — то, чего нет в самом iiko.

**Главные особенности**:
- Режим **read-only**: модуль только читает и отображает данные из iiko, управлять стоп-листом через него нельзя
- **Многослойное кэширование** (4 кэша с разными TTL)
- **Расчёт упущенной выручки** (Potential Lost Revenue) на основе исторических данных продаж за настраиваемый период
- **Деградированный режим**: система работает даже при частичной недоступности источников данных

**Главные риски и проблемы**:
1. Матчинг выручки по имени блюда (нечёткий, зависит от совпадения строки) — ключевая методологическая слабость
2. Отсутствие истории и аудита — только снимок «сейчас»
3. Отсутствие алертинга — данные есть, но нет реакции системы на критические события
4. Нет возможности управления стоп-листом из дашборда
5. Всё состояние — in-memory, перезапуск сервера обнуляет кэши

# 2. ЧТО ИЗ СЕБЯ ПРЕДСТАВЛЯЕТ МОДУЛЬ

Модуль «Стоп-лист» — это операционный отчёт в реальном времени, отвечающий на вопрос: **«Какие позиции меню прямо сейчас недоступны на кассах — и сколько мы из-за этого теряем?»**

В ресторанном бизнесе «стоп-лист» — это список блюд/модификаторов/групп, которые кассир или менеджер временно отключил на терминале: закончились ингредиенты, блюдо снято с продажи, технологический перерыв. Управление происходит непосредственно в iiko (на кассах или в back-office iiko), а данный модуль является **витриной аналитики** поверх этих данных.

**Объекты контроля**: три типа сущностей меню iiko:
- `product` — конкретное блюдо
- `modifier` — модификатор (дополнение, соус, вариант приготовления)
- `group` — группа позиций меню

**Место в бизнес-процессе**: модуль встраивается в операционный контроль — между фактом остановки продаж в iiko и управленческим решением по восстановлению доступности.

# 3. ЗАЧЕМ НУЖЕН МОДУЛЬ

**Бизнес-проблема**: В нативном интерфейсе iiko стоп-лист представлен отдельно по каждому терминалу и не агрегирован по сети. Менеджер или аналитик не видит:
- сколько позиций в стопе прямо сейчас по всей сети
- как давно они там находятся
- сколько денег теряет ресторан каждый час

**Риски без модуля**:
- Позиция может неделями находиться в стопе незамеченной (зафиксирован счётчик `longerThan1Day`)
- Нет приоритизации: какую позицию важнее всего вернуть в продажу
- Нет финансовой оценки простоя — невозможно обосновать срочность действий

**Польза для бизнеса**:
- Единое окно мониторинга по всем организациям и терминалам
- Финансовая мотивация для оперативного реагирования (PLR — Potential Lost Revenue)
- Ранжирование позиций по приоритету (длительность × выручка)

# 4. КАК РАБОТАЕТ МОДУЛЬ ПОШАГОВО

### Шаг 1 — Инициализация запроса (Frontend → Backend)
Пользователь открывает страницу `/stop-list`. Pinia store `useStopListStore` сначала загружает список организаций через `organizationsApi`, затем вызывает `GET /api/stop-lists?organizationId=...&timezone=...`.

### Шаг 2 — Валидация и разбор параметров (Controller)
Controller проверяет:
- наличие `organizationId` (обязательный)
- длину `timezone` (≤ 100 символов)
Передаёт запрос в `stopListService.getStopLists(query)`.

### Шаг 3 — Проверка нормализованного кэша (Service)
Service строит `cacheKey = sorted(organizationIds) + "::" + timezone`. Если кэш свежий (TTL = 20 сек) и `refresh !== true` — возвращает кэшированный результат немедленно.

### Шаг 4 — Параллельный сбор данных из iiko (IikoService)
Запускаются три независимых запроса (последовательно по зависимостям, параллельно по организациям):

```
[1] fetchAccessToken()          → Bearer-токен (кэш 9 мин)
     ↓
[2] fetchStopListsWithProducts() → сырые данные стоп-листа (кэш 45 сек)
[3] fetchMenuV2Index()          → индекс меню: id→name (кэш 10 мин)
[4] _buildRevenuePerHourIndex() → индекс продаж по блюдам (через topDishes)
```

Шаги [2], [3], [4] выполняются параллельно после получения токена.

### Шаг 5 — Нормализация сырых данных iiko
`normalizeStopListItems()` в IikoService рекурсивно обходит структуру `terminalGroupStopLists` из ответа iiko, наследует контекст (organizationId, terminalGroupId) вниз по дереву и собирает плоский список записей с полями: `id`, `entityType`, `entityId`, `entityName`, `isInStop`, `startedAt`, `endedAt`, `balance`, `reason`.

Ключевой момент: **`isInStop` выводится эвристически** — если `endedAt` отсутствует, позиция считается активной.

### Шаг 6 — Обогащение данных (Enrichment pipeline)
```
normalizeItems()           → унификация типов, вычисление inStopMinutes/Hours/Days
enrichEntityNames()        → подстановка названий из меню v2 / номенклатуры (если iiko вернул пустое имя)
organizationName patch     → подстановка имени организации из локального справочника
enrichEstimatedLostRevenue() → расчёт PLR для активных позиций с известным avgRevenuePerHour
```

### Шаг 7 — Дедупликация и сортировка
Дедупликация по составному `id = orgId:terminalGroupId:entityType:entityId`. При дублях выигрывает запись с более поздним `startedAt`. Сортировка: по `startedAt DESC`.

### Шаг 8 — Формирование ответа и кэширование
Service собирает `meta` (счётчики, warnings, источник, период PLR) и кэширует результат на 20 секунд. Frontend получает `{ data: { items, organizations }, meta }`.

### Шаг 9 — Клиентская фильтрация
Pinia store применяет фильтры на клиенте: по организации, terminal group, типу сущности, статусу (active/completed/all), длительности (gt1h/gt2h/gt24h), полнотекстовому поиску по имени/id/причине.

# 6. БИЗНЕС-ЛОГИКА

### Правила попадания в стоп-лист
Стоп-лист формируется **исключительно в системе iiko** — через кассу, back-office или API iiko. Данный модуль не управляет этим процессом. Причины попадания (со стороны бизнеса): закончился товар, блюдо временно убрано, технологический перерыв, ошибка персонала.

### Правила определения статуса «активен»
Эвристика в `_collectNormalizedItemFromNode`:
```js
isInStop = endedAt ? false : node.isInStop === false ? false : node.isInStopList === false ? false : true
```
**Логика**: если `endedAt` заполнен — стоп завершён. Иначе — смотрим явные флаги iiko. Если ни того ни другого нет — считаем активным. Это «оптимистичная» логика: **неизвестное = активное**.

### Правила расчёта PLR (Potential Lost Revenue)
```
estimatedLostRevenue = avgRevenuePerHour × inStopHours
avgRevenuePerHour = totalRevenue(N дней) / totalHours(N дней)
```
Условия начисления PLR:
- `isInStop === true` (только активные)
- `inStopHours > 0`
- `avgRevenuePerHour > 0` (блюдо есть в истории продаж за период)
- Матчинг по `entityName.toLowerCase()` ↔ `dish.name.toLowerCase()` из topDishes

**Критический момент**: матчинг только по имени, без ID. Если в iiko называется «Борщ», а в topDishes «борщ домашний» — PLR не рассчитается.

### Правила дедупликации
Одна и та же позиция (по составному ID) может присутствовать несколько раз, если iiko вернул дубли. Побеждает запись с наибольшим `startedAt`. Это означает, что **повторное добавление в стоп скрывает предыдущую запись** — история теряется.

### Правила кэширования (многоуровневые)
| Уровень | TTL | Стратегия инвалидации |
|---|---|---|
| Token | 9 мин | `forceRefresh` или истечение |
| Raw stop-lists | 45 сек | `forceRefresh` или истечение |
| Menu v2 / Nomenclature | 10 мин | `forceRefresh` или истечение |
| Normalized result | 20 сек | `forceRefresh` или истечение |

`refresh=true` от пользователя сбрасывает **все** кэши через цепочку вызовов.

# 7. USER FLOWS / SYSTEM FLOWS

### Сценарий 1 — Открытие страницы стоп-листа

**Триггер**: менеджер переходит на `/stop-list`  
**Шаги**:
1. Store загружает список организаций → GET `/api/organizations`
2. Store вызывает `loadStopLists()` → GET `/api/stop-lists?organizationId=...`
3. Backend проверяет кэш → если свежий, возвращает сразу (< 20 сек)
4. Если нет — выполняет полный pipeline (iiko auth → stop-lists → menu → PLR)
5. Frontend рендерит счётчики, KPI и таблицу
**Результат**: актуальный снимок стоп-листа  
**Проблемы**: при первом открытии (нет кэша) — суммарный латенс может быть 3–8 сек из-за цепочки запросов к iiko

### Сценарий 2 — Принудительное обновление

**Триггер**: пользователь нажимает кнопку «Обновить» в `ReportPageHeader`  
**Шаги**:
1. Frontend отменяет текущий запрос через `AbortController`
2. Вызывает `loadStopLists({ refresh: true })`
3. Backend получает `refresh=true` → сбрасывает все кэши → полный pipeline
**Результат**: свежие данные из iiko  
**Проблемы**: если iiko медленно отвечает — пользователь ждёт без внятного прогресса

### Сценарий 3 — Фильтрация

**Триггер**: пользователь меняет фильтр (organization, terminal group, status, duration, entityType, search)  
**Шаги**: только клиентская фильтрация `filteredItems` через геттер Pinia  
**Результат**: мгновенный отклик без запросов к серверу  
**Проблемы**: при большом количестве позиций — геттер может тормозить (нет пагинации, нет мемоизации)

### Сценарий 4 — Деградированный режим (topDishes недоступен)

**Триггер**: `topDishesService.getDishesDataset()` бросает исключение  
**Шаги**:
1. `_buildRevenuePerHourIndexByOrganization` перехватывает ошибку
2. Записывает warning, возвращает пустой Map для организации
3. `enrichEstimatedLostRevenue` не находит совпадений → `estimatedLostRevenue = null`
**Результат**: стоп-лист отображается, PLR = null, в хедере появляется warning  
**Итог**: корректная деградация без падения

### Сценарий 5 — Переключение организации

**Триггер**: пользователь меняет организацию в фильтре  
**Шаги**:
1. `setCurrentOrganization(id)` → `loadStopLists()`
2. Формируется новый `organizationId` query
3. Старый запрос отменяется через `AbortController`
4. Backend проверяет кэш по новому ключу
**Проблемы**: timezone resolves к timezone первой найденной организации; если организация не найдена в списке — используется `Intl.DateTimeFormat` браузера

# 8. РОЛИ И ПРАВА ДОСТУПА

**Текущая реализация**: система ролей **отсутствует**. Любой авторизованный пользователь дашборда имеет одинаковый полный доступ к данным стоп-листа.

**Что потенциально нужно разграничить**:

| Действие | Кто должен иметь доступ |
|---|---|
| Просмотр стоп-листа | Менеджеры, аналитики, операторы |
| Принудительное обновление | Менеджеры и выше |
| Просмотр финансовых KPI (PLR) | Аналитики, руководство |
| Управление стоп-листом (снятие позиций) | Отсутствует в системе |

**Риски**:
- Финансовые метрики (PLR, avgRevenuePerHour) видны всем — данные о выручке на блюдо могут быть чувствительными
- Принудительный `refresh` доступен всем — теоретически можно нагрузить iiko API частыми обновлениями
- Нет журнала действий пользователей

# 9. СОСТОЯНИЯ, СТАТУСЫ И ПЕРЕХОДЫ

### Статусы позиции стоп-листа

```
[Активен в стопе]  ──── endedAt появился ────▶  [Завершён]
  isInStop: true                                   isInStop: false
  endedAt: null                                    endedAt: ISO string
  
[Данные неполные]
  startedAt: null
  inStopHours: null
  isInStop: true (по умолчанию)
```

**Проблема**: нет промежуточных состояний. Не поддерживаются: «приостановлен», «ожидает подтверждения», «автоматически снят». Все эти нюансы iiko теряются при нормализации — система сводит богатую модель iiko к бинарному `isInStop`.

### Состояния кэша

```
[Нет кэша] → [Заполняется] → [Свежий] → [Устаревший]
                                         ↑____TTL истёк____↑
```

При `refresh=true` — прямой переход `[Свежий] → [Нет кэша] → [Заполняется]`.

**Конфликт состояний**: 4 разных кэша с разными TTL могут находиться в разной степени свежести. Теоретически возможна ситуация: `rawStopLists` уже обновился (45 сек), а `menuV2Index` ещё нет (10 мин) → позиции есть, но названия старые.

# 10. ВАЛИДАЦИИ И ОГРАНИЧЕНИЯ

### Что проверяется (Backend)
- Наличие `organizationId` (400 если отсутствует)
- Длина `timezone` (400 если > 100 символов)
- Диапазон `lostRevenueLookbackDays` (3–60, иначе default 14)

### Что НЕ проверяется — критические пробелы
1. **Формат `organizationId`** — принимаются любые строки, включая SQL-инъекции (хотя они идут напрямую в iiko API, это снижает риск, но не исключает его полностью)
2. **Валидность `timezone`** — строка до 100 символов не проверяется на существование в базе IANA. `timezone = "invalid/zone"` уйдёт в iiko и может вызвать непредсказуемый ответ
3. **Rate limiting** — нет ограничения частоты запросов. Параметр `refresh=true` можно вызывать без ограничений
4. **Размер ответа** — нет лимита на количество позиций в ответе, нет пагинации

### Клиентские ограничения
- Поиск работает только по уже загруженным данным (фильтрация на клиенте)
- При переключении организации — весь список перезагружается, нет incremental fetch

# 11. ОШИБКИ И EDGE CASES

### Задокументированные ошибки
| Код | HTTP | Условие |
|---|---|---|
| `VALIDATION_ERROR` | 400 | Нет `organizationId` или `timezone` слишком длинный |
| `UPSTREAM_ERROR` | 502 | `IikoApiError` от iiko API |
| `INTERNAL_ERROR` | 500 | Непредвиденная ошибка в service |

### Edge cases, требующие внимания

**1. Позиция в стопе без `startedAt`**
iiko API не всегда возвращает дату начала. Результат: `inStopHours = null`, `estimatedLostRevenue = null`. Позиция отображается, но PLR не считается. Warning в meta. **Риск**: позиции могут быть в стопе неопределённо долго — невидимые для метрик.

**2. Одна позиция на нескольких терминалах одновременно**
Составной ID включает `terminalGroupId`, поэтому одно блюдо на 5 терминалах = 5 отдельных записей с отдельным PLR каждая. Суммарный PLR = сумма по всем записям. Это корректно с точки зрения бизнеса (5 терминалов = 5 упущенных потоков выручки), но может пугать пользователя суммой.

**3. Позиция повторно добавлена в стоп (добавили, сняли, добавили снова)**
Дедупликация оставляет **только последнюю запись** по `startedAt`. Предыдущий эпизод в стопе теряется. Это корректно для операционного мониторинга «сейчас», но некорректно для аналитики «сколько раз блюдо уходило в стоп».

**4. PLR с именем, которого нет в топ-блюдах**
Если блюдо недавно добавлено в меню и ещё не продавалось в период lookback — `avgRevenuePerHour = null`, PLR не считается. Система корректно молчит, но пользователь не понимает, почему у позиции нет PLR.

**5. `endedAt < startedAt` (битые данные из iiko)**
```js
if (endTs < startTs) {
  return { inStopMinutes: null, inStopHours: null, inStopDays: null };
}
```
Защита есть — возвращаются `null`. Хорошо, но warning не генерируется.

**6. Перезапуск backend-сервера**
Все кэши (token, rawStopLists, nomenclature, menuV2, normalizedCache) — in-memory. При рестарте — первый запрос «холодный», latency высокий.

**7. Одновременные запросы на `refresh` от разных пользователей**
Нет mutex или lock на операцию обновления кэша. Два параллельных `refresh=true` — два одновременных полных pipeline запросов к iiko. Потенциально нагружает iiko и удваивает время ответа.

**8. `organizationId` не существует в iiko**
iiko вернёт пустой массив или ошибку — обработается как пустой стоп-лист или `IikoApiError`. Пользователь увидит либо пустую таблицу, либо ошибку 502 — без объяснения причины.

**9. Срабатывание fallback на nomenclature вместо menu v2**
Если menu v2 возвращает пустой индекс — система тихо fallback-ится на `/nomenclature`. Пользователь не знает об этом. Nomenclature — более старый и медленный endpoint iiko.

**10. Отмена запроса (AbortController)**
Если пользователь быстро переключает организации, старые запросы отменяются — правильно. Но если отменяется запрос, который уже получил данные и пишет в state — `isLoading` может зависнуть из-за проверки `currentRequestId !== this.requestId`.

# 12. ИНТЕГРАЦИИ И ЗАВИСИМОСТИ

### Зависимости модуля

```
stop-list/service
    ├── organizations/service      → список организаций, timezone
    ├── stop-list/iikoService      → данные из iiko API (stop-lists, token, menu/nomenclature)
    └── topDishes/service          → исторические данные продаж для PLR
```

### Влияние на другие модули
Модуль **потребляет** `topDishes/service`, что создаёт скрытую нагрузку: каждый запрос стоп-листа запускает запрос к topDishes (с собственным кэшем). При кэш-промахе — обоих кэшей — суммарный latency значительно вырастает.

### Внешние зависимости

| Зависимость | Критичность | Fallback |
|---|---|---|
| iiko Cloud API (`/stop_lists`) | Критическая | Нет (502 пользователю) |
| iiko Cloud API (`/access_token`) | Критическая | Нет (кэш 9 мин) |
| iiko Cloud API (`/api/2/menu`) | Умеренная | Fallback на `/nomenclature` |
| iiko Cloud API (`/nomenclature`) | Умеренная | Пустые имена (fallback «Без названия») |
| topDishes service | Некритическая | PLR = null, warning в meta |

### Риски синхронизации
- Меню в iiko может обновиться, а кэш nomenclature ещё свежий (10 мин) → новые блюда будут «Без названия» до истечения кэша
- `orgNamesById` строится из локального списка organizations, но iiko оперирует своим ID — рассинхронизация невозможна только при условии, что organizations/service и iiko синхронизированы

# 13. СИЛЬНЫЕ СТОРОНЫ РЕАЛИЗАЦИИ

**1. Деградация без падения**
Ошибки в расчёте PLR (topDishes недоступен), fallback с menu v2 на nomenclature — всё обрабатывается gracefully с warnings в meta. Система не падает при частичной недоступности.

**2. AbortController на фронтенде**
Гонки запросов при быстром переключении фильтров исключены. requestId-паттерн дополнительно защищает от применения устаревших данных.

**3. Многоуровневый fallback при получении имён сущностей**
`menu v2 by_id (with externalMenuId) → menu v2 (organizationIds) → menu v2 (first org) → nomenclature (batch) → nomenclature (per org)` — 5 попыток перед тем, как позиция получит «Без названия».

**4. Нормализация `isInStop` с несколькими fallback-полями iiko**
iiko API в разных версиях возвращает разные поля (`isInStop`, `isInStopList`, presence of `endedAt`). Код корректно обрабатывает все варианты.

**5. Строгая типизация при нормализации**
Все поля приводятся к ожидаемым типам с явными fallback-значениями: `String(value || "")`, `Number.isFinite()` guards, белые списки для `entityType`. Риск runtime-ошибок от неожиданных данных iiko минимален.

**6. `refresh` параметр сквозной через всю цепочку**
Пользовательский `refresh=true` доходит до каждого кэша и корректно инвалидирует все уровни.

**7. Составной ID для дедупликации**
`orgId:terminalGroupId:entityType:entityId` — правильно учитывает, что одно блюдо на разных терминалах — это разные события. Без `terminalGroupId` дедупликация была бы некорректной.

# 14. СЛАБЫЕ СТОРОНЫ И РИСКИ

**1. Матчинг PLR по имени блюда — методологически слабое место**
`avgRevenuePerHour` привязывается к позиции по `entityName.toLowerCase()`. Если iiko вернул «Борщ», а в topDishes записано «Борщ домашний» — матчинг не сработает. Нет ID-based связки. Процент позиций без PLR может быть очень высоким в реальных данных.

**2. Отсутствие истории**
Модуль показывает только текущий снимок. Нет записи о том, что позиция была в стопе вчера 4 часа, позавчера 2 часа. Аналитика динамики невозможна. `feature-readiness-matrix.md` сам подтверждает: «История статусов частичная».

**3. Нет алертинга**
Позиция может находиться в стопе 3 дня, PLR накапливается, а система не сигнализирует ответственному лицу. Бот (bot) не интегрирован со стоп-листом вообще.

**4. Нет управления из дашборда**
Пользователь видит проблему, но не может её устранить через интерфейс. Для снятия позиции со стопа нужно идти в iiko отдельно. Это разрывает операционный цикл.

**5. Все кэши in-memory, без персистентности**
Перезапуск Node.js — полный сброс. В production с частыми рестартами (деплои, краши) это означает регулярные «холодные» старты с высоким latency.

**6. Потенциальная проблема concurrent refresh**
Два одновременных запроса с `refresh=true` — два независимых полных pipeline. Нет защиты от дублирования (mutex, promise deduplication). В iiko API могут быть rate limits.

**7. Линейная модель PLR не учитывает время суток**
`PLR = avgRevenue/hour × hours` предполагает равномерное распределение выручки. Блюдо в стопе с 14:00 до 16:00 (непиковое время) и с 19:00 до 21:00 (ужин) даст одинаковый PLR. В ресторанном бизнесе это значительная погрешность.

**8. `balance` поле передаётся, но не используется**
В normalizeItems и в UI есть колонка «баланс», но значение `balance` из iiko (количественный остаток) не используется в бизнес-логике. Непонятно, нулевой баланс = в стопе?

**9. Нет валидации формата `timezone` и `organizationId`**
Принимаются произвольные строки.

# 16. РЕКОМЕНДАЦИИ ПО УЛУЧШЕНИЮ

### 1. Перейти на ID-based матчинг для PLR
**Что изменить**: в `topDishes/service` добавить возврат `entityId` (productId из iiko) вместе с именем. В `enrichEstimatedLostRevenue` матчинг делать по `entityId`, с fallback на имя.  
**Почему**: текущий name-based матчинг даёт систематически заниженный охват PLR, что делает ключевую метрику модуля недостоверной.  
**Что даст**: достоверный PLR для большинства позиций, устранение ложных пропусков.  
**Сложность**: MEDIUM | **Приоритет**: HIGH

### 2. Добавить алертинг в Telegram-бота
**Что изменить**: в bot создать scheduler, который периодически вызывает стоп-лист API и отправляет уведомление при: позиция в стопе > 2 часов И `estimatedLostRevenue > threshold` ИЛИ позиция в стопе > 24 часов.  
**Почему**: пассивный мониторинг не работает в операционном контексте. Нужен проактивный сигнал ответственному лицу.  
**Что даст**: сокращение среднего времени в стопе, снижение фактического PLR.  
**Сложность**: LOW | **Приоритет**: HIGH

### 3. Добавить персистентность истории (снапшоты)
**Что изменить**: ввести периодическое сохранение снапшотов стоп-листа в БД (или JSON-файл как `plans.json`). Хранить: `startedAt`, `endedAt`, `entityId`, `orgId`, `estimatedLostRevenue` на момент снятия.  
**Почему**: операционный анализ «почему X было в стопе 3 дня» невозможен без истории.  
**Что даст**: аналитика динамики, heatmap проблемных позиций, отчёты за период.  
**Сложность**: MEDIUM | **Приоритет**: HIGH

### 4. Защита от concurrent refresh (promise deduplication)
**Что изменить**: в `StopListService.getStopLists()` добавить в-полёт промис: если запрос с `refresh=true` уже выполняется — вернуть тот же промис, а не запускать новый.  
**Почему**: два одновременных refresh — двойная нагрузка на iiko без пользы.  
**Что даст**: защита от случайной перегрузки iiko, снижение latency при параллельных запросах.  
**Сложность**: LOW | **Приоритет**: MEDIUM

### 5. Почасовая корректировка PLR
**Что изменить**: в индексе выручки хранить почасовое распределение (`revenue[hour_of_day]`). PLR считать как сумму `avgRevenue[hour]` для каждого часа нахождения в стопе.  
**Почему**: блюдо в стопе в 14:00 vs 19:00 — разный ущерб.  
**Что даст**: более точная оценка ущерба, лучшая приоритизация действий.  
**Сложность**: HIGH | **Приоритет**: MEDIUM

### 6. Валидация `timezone` и `organizationId`
**Что изменить**: добавить проверку timezone через `Intl.supportedValuesOf('timeZone')` или allowlist. Добавить regex для formata UUID на `organizationId`.  
**Почему**: текущее отсутствие валидации передаёт arbitrary strings во внешний API.  
**Что даст**: предсказуемые ошибки вместо неожиданного поведения iiko.  
**Сложность**: LOW | **Приоритет**: MEDIUM

### 7. Интеграция `balance` в определение `isInStop`
**Что изменить**: если iiko возвращает `balance = 0` — явно проставлять `isInStop = true` независимо от флагов.  
**Почему**: нулевой остаток = фактически в стопе, даже если флаг iiko не выставлен.  
**Что даст**: более точный список активных стопов.  
**Сложность**: LOW | **Приоритет**: LOW

## Общий вердикт

Модуль «Стоп-лист» — **технически добросовестная реализация операционного read-only дашборда**, построенная поверх сложного и нестабильного внешнего API (iiko). Разработчик правильно справился с ключевой технической задачей: превратить неудобные raw-данные iiko в читаемый интерфейс с деградацией при недоступности источников.

**Что сделано хорошо**: многоуровневый fallback при получении данных, graceful degradation, AbortController на клиенте, строгая нормализация входящих данных от iiko, составной ID для корректной дедупликации.

**Что тянет вниз**: PLR-метрика (главная бизнес-ценность модуля) работает ненадёжно из-за name-based матчинга — это подрывает доверие к ключевому KPI. Отсутствие истории и алертинга превращает модуль из инструмента управления в **пассивный монитор**, который показывает проблему, но не помогает на неё реагировать.

**Что исправить в первую очередь**: перейти на ID-based матчинг PLR + добавить алертинг в бота. Эти два изменения кратно увеличат практическую ценность модуля при относительно невысоких затратах.
